## 1. 产品定位与用户分析

### 1.1 项目是什么

**数智兴安 (szxa-next)** 是面向工程施工现场的 **数字化管控平台**。以智能安全帽为核心 IoT 终端，覆盖人员定位追踪、设备管理、安全风险管控、考勤、物资出入库、环境监测、视频监控等业务域。C/S + B/S 混合架构，支持桌面端（Tauri）、Web 端、移动端（Expo）三端。

### 1.2 面向的客户与场景

| 维度 | 说明 |
|------|------|
| **目标行业** | 工程施工 / 基础设施建设（如水电、路桥、房建等大型工程项目） |
| **客户类型** | 工程总承包企业（如中南院）、二级施工单位、项目部 |
| **典型规模** | 单个项目部数百至数千名施工人员，数十至上百台安全帽设备，多个仓库/存放点 |
| **使用人数** | 按项目隔离，单项目在线用户数十人（管理员 + 班组长 + 负责人），并发请求以管理操作为主 |
| **使用时长** | 项目全生命周期（数月到数年），7×24 小时设备数据采集，工作日白天管理操作密集 |
| **网络环境** | 施工现场网络不稳定，桌面端需支持离线使用 + 在线同步 |

### 1.3 组织层级模型

系统按 **租户 → 项目** 两级隔离，组织树从公司级到班组共 7 层：

```
公司级 (Company)
  └── 二级单位 (SecondaryUnit)
        └── 项目部 (ProjectDepartment)  ← 项目数据隔离边界
              ├── 项目参建单位 (ProjectParticipant)
              │     └── 班组 (Team)
              ├── 部门 (Department)
              └── 科室 (Office)
```

每个项目独立配置模块、权限、数据，项目间数据不可互访。用户必须先选择项目才能进入功能页面。

### 1.4 用户角色全景

系统存在**两类权限模型**，分别服务于不同目的：

**A. 操作权限（RBAC）** — 控制"谁能看到什么页面、能做什么操作"

```
用户 → 操作角色（OperationRole）→ 权限条目（UserVsPermission）
                                    ├── PageModule (模块级可见性)
                                    ├── PageMenu (菜单级可见性)
                                    ├── CommonAction (Read/Write/Delete)
                                    └── PageAction (页面按钮级操作)
```

**B. 数据权限** — 控制"用户在项目内能看到哪些数据"

```
用户 → 数据角色（DataRole）→ 数据约束（DataConstraint）
                                ├── BooleanConstraint
                                ├── EnumerableDataConstraint
                                ├── NumericDataConstraint（大于/小于/等于）
                                └── EnumSetConstraint
```

**C. 业务角色（非系统强枚举，来自业务 spec 中的角色定义）**

| 业务角色 | 典型职责 | 使用模块 |
|---------|---------|---------|
| **HSE 管理员**（安全管理员） | 维护材料分类字典、仓库信息、出入库流水；管理安全风险库、隐患排查 | 物资安全、安全风险、隐患排查 |
| **仓储保管人** | 登记材料入库/出库流水 | 物资安全（出入库） |
| **项目负责人/项目经理** | 查看库存总览、存储点 GIS 分布、监控视频、项目统计 | 项目总览、物资安全、监控管理 |
| **环境监测负责人** | 设置告警规则、查看与处理告警、配置显示单位 | 环境监测 |
| **设备配置管理员** | 创建维护设备接口、管理监测设备台账、同步日志 | 环境监测、设备管理 |
| **班组长** | 组织班前会、查看本班组人员状态 | 智能班前会、现场管理 |
| **普通用户/施工人员** | 查看实时监测数据、个人信息、接收广播 | 现场管理、人员管理 |
| **系统管理员** | 用户账号管理、权限分配、项目管理、租户管理 | 后台管理 |
| **系统**（自动化） | 定时数据同步、告警判断、日志记录、ETL 管道 | DataCenter.ETL |

> **注意**：业务角色是概念层的划分（来自各模块需求 spec），具体权限控制通过操作角色+数据角色组合实现，不同项目可为同一用户分配不同角色。

### 1.5 三端差异化需求

| 端 | 用户 | 核心场景 | 特有需求 |
|----|------|---------|---------|
| **桌面端** (Tauri) | 项目部管理人员的办公电脑 | 数据录入、GIS 地图查看、报表导出、后台管理 | 离线可用、本地 SQLite、GIS 离线瓦片 |
| **Web 端** (浏览器) | 任意可联网设备 | 同桌面端功能，但依赖网络 | 无需安装，轻量访问 |
| **移动端** (Expo) | 现场施工人员、班组长 | 扫码绑设备、声纹录入、证件拍照上传、接收告警推送 | 拍照、扫码、推送通知、简化交互 |

### 1.6 访问量与负载估算

> 以下为基于代码中已有配置和业务常识的**量级估算**（非精确测量），用于技术选型和容量规划的参考。

#### 请求分类

系统负载分为三种截然不同的流量模型：

```
┌──────────────────────────────────────────────────────────────┐
│  类型A: IoT 数据流（持续高频，7×24）                            │
│  安全帽 → RabbitMQ → Consumer → PostgreSQL                    │
│  ~3.3 TPS（电量） + 位置上报 + 心跳 + 告警 ≈ 10~20 TPS 合计     │
│  单项目数百设备 × 每设备数秒上报一次                              │
├──────────────────────────────────────────────────────────────┤
│  类型B: 管理操作（低频突发，工作日白天）                          │
│  浏览器 → REST API → PostgreSQL                                │
│  并发用户数：单项目 10~50 人（管理员 + 班组长 + 负责人）           │
│  操作频率：每人每分钟 1~5 次 CRUD，峰值可能翻倍                   │
│  预估 QPS：稳态 < 10，峰值 < 50                                 │
├──────────────────────────────────────────────────────────────┤
│  类型C: 实时推送（持续连接，少量写入）                            │
│  后端 SignalR → 前端 WebSocket                                  │
│  连接数：单项目在线用户数（10~50）                                │
│  推送频率：人员位置变化时（按设备上报频率触发）                     │
└──────────────────────────────────────────────────────────────┘
```

#### 按数据来源估算

| 数据来源 | 触发方式 | 频率/量级 | 代码证据 |
|---------|---------|----------|---------|
| **安全帽电量** | RabbitMQ 连续消费 | **~3.3 TPS**（单项目），批量刷库 BatchSize=200 | `DeviceBatteryConsumerService.cs` 注释 `~3.3 TPS` |
| **安全帽位置** | RabbitMQ 连续消费 | 数百设备 × 每 3~10 秒/次上报 ≈ **30~100 TPS** | `DeviceLocationConsumerService` |
| **安全帽心跳** | RabbitMQ 连续消费 | 每设备每 30~60 秒一次 ≈ **5~15 TPS** | `DeviceHeartbeatConsumerService` |
| **安全帽告警** | RabbitMQ 连续消费 | 偶发（跌倒/SOS/静默），常态 < 1 TPS | `DeviceAlarmConsumerService` BatchSize=200 |
| **环境监测同步** | Cron 定时拉取 | 按 `SyncIntervalSeconds` 配置（通常 5~15 分钟），拉取量取决于外部平台设备数 | `EnvCollectScheduler` + `SyncIntervalSeconds` |
| **视频流** | WebSocket 直连 | 按需播放，单路流 ~1 Mbps；不经过后端 API | 前端 → 视频平台直连 |
| **管理 CRUD** | HTTP REST API | 稳态 < 10 QPS，峰值 < 50 QPS（单项目） | 17 个模块合计 |
| **人员位置推送** | SignalR Hub | 每位置批次 → 广播到在线客户端 | `/hubs/worker-location` |
| **GIS 瓦片** | HTTP GET | 浏览器缓存 + 本地离线瓦片；在线时按视口加载 | 天地图 / 离线 .pak |
| **文件上传** | HTTP Multipart | 偶发（证件照片、体检报告、方案文件），单文件 < 100 MB | FileStorage（Local/S3） |

#### 多项目叠加

系统是多项目隔离的。假设平台上有 N 个活跃项目：

| 流量类型 | 单项目量级 | N=10 项目 | N=50 项目 |
|---------|-----------|----------|----------|
| IoT 数据流合计 | 50~130 TPS | 500~1300 TPS | 2500~6500 TPS |
| 管理 API 合计 | < 50 QPS | < 500 QPS | < 2500 QPS |
| SignalR 连接数 | 10~50 | 100~500 | 500~2500 |
| 数据库连接池 | ~20 | ~50 | ~100+ |

#### 关键性能路径（NFR 热点）

| 路径 | 为什么是热点 | 当前设计 |
|------|------------|---------|
| **库存台账查询** | 材料库存总览需 JOIN `material_inventory_ledger` + `materials`，分页+排序 | 物化台账直接分页（NFR-001），代码注释标注为关键性能路径 |
| **出入库事务** | 入库/出库需在同一事务内完成：台账增减 + 存放量增减 + 主表插入 + 明细插入 | 纯 Npgsql 事务，避免 EF Core 变更追踪开销 |
| **人员位置批量写入** | 数百设备 × 高频上报 | `PersonLocationBuffer` 批量写入 |
| **电量批量写入** | ~3.3 TPS 持续流入 | `ConcurrentQueue` 缓冲 + BatchSize=200 + 定时刷新 |
| **GIS 地图标记** | 数百人员标记同时渲染 | 前端增量更新（added/updated/removed），仅更新变化标记 |
| **离线同步** | 桌面端同步全量基础数据 | `BatchJsonDataSynchronizer`，5 秒延迟启动 |

#### 容量规划建议

- **数据库**：IoT 写入是主要压力源（持续高频）。单项目 50~130 TPS 写入，PostgreSQL 单实例可轻松承载。多项目 50+ 时考虑读写分离或时序库。
- **RabbitMQ**：IoT 消息中枢，需保证不丢消息（电量可丢、告警不可丢）。当前设计已区分：电量/位置 ack 即丢弃，告警需可靠处理。
- **SignalR**：连接数不是瓶颈（单项目 10~50），但推送频率需控制——不要在每一条位置消息时都广播，应做批量/节流。
- **API 服务器**：管理操作负载很低（< 50 QPS），单实例足够。真正的瓶颈在 IoT 消费端的处理能力。
- **前端**：GIS 地图渲染是客户端瓶颈（数百标记），已通过增量更新 + 视口裁剪优化。

### 3.5 架构核心概念：模块功能 vs 切面功能

理解这个项目的架构，最关键的一个心智模型是区分两种不同性质的功能：

```
┌────────────────────────────────────────────────────────────┐
│                      切面功能（横向）                        │
│  权限 · 项目隔离 · 时区 · 认证 · 日志 · 异常 · ID生成 ·     │
│  统一响应 · Swagger · 数据库规范 · 集成事件 · 文件存储      │
│  每个模块都要遵守，改一处影响全部模块                          │
├────────────────────────────────────────────────────────────┤
│  模块A        模块B        模块C        模块D       ...     │
│  (现场管理)   (人员管理)   (物资安全)   (环境监测)          │
│                                                             │
│                    模块功能（纵向）                           │
│  具体业务逻辑、实体、API端点、页面、ViewModel                 │
│  每个模块独立，改一个模块不影响其他                            │
└────────────────────────────────────────────────────────────┘
```

- **模块功能**（纵向）：每个业务模块自己的领域逻辑——实体、Service、Controller、ViewModel、前端页面。改模块 A 不影响模块 B。
- **切面功能**（横向）：横跨所有模块的基础设施和约束——权限、项目隔离、时区、认证、日志、统一响应格式……它们不是某个模块的私有财产，而是**每个模块都必须遵守的契约**。**切面的改动影响全局，需要更加谨慎**。

> ⚠️ **切面改动警示**：改一个切面（如新增 PageModule 枚举值、修改 ApiResult 格式、调整项目隔离逻辑），可能影响全部 17 个模块。切面变更必须评估全局影响，不能只考虑当前正在开发的模块。

### 3.6 新建模块检查清单

以下按"模块功能"与"切面功能"分开列出。每个模块只需关注自己的模块功能，但**必须保证与所有切面正确集成**。

#### A. 模块功能（每个模块独立完成）

| # | 事项 | 位置/方式 | 类别 |
|---|------|----------|------|
| 1 | **创建模块项目** | `{Module}.API/` 目录，参考现有 csproj 结构 | 模块 |
| 2 | **实现 IModule 接口** | `{Module}Module.cs`，继承 `Module` 基类 | 模块 |
| 3 | **在 ModulesLoader 注册** | `HostAPP/ModulesLoader.cs` → `_modules` 列表 | 模块 |
| 4 | **创建唯一 Service** | `Business/Services/{Module}Service.cs`（架构约束 §21：每模块仅 1 个 Service） | 模块 |
| 5 | **创建 Repository** | `Repository/`，继承 `DirectKeyRepositoryBase` 或 `RelationRepositoryBase` | 模块 |
| 6 | **创建 Controller** | `Controllers/`，RPC 风格：`[HttpPost] C/U/D/List` + `[HttpGet] Item` | 模块 |

## 7. 关键架构决策与约束

### 7.1 核心 ADR（架构决策记录）

| ADR | 决策 | 影响 |
|-----|------|------|
| ADR-1 | RxJS Subject 事件总线，不用 Redux/Zustand | 前端跨组件通信用 `GlobalEvents` |
| ADR-2 | OpenAPI Generator 自动生成 API 客户端 | `src/clients/` 禁止手动修改 |
| ADR-3 | GIS 用 Manager 模式（非全部在组件内） | MarkerManager/FenceManager/TrailManager 等 |
| ADR-4 | 平台抽象层（接口+工厂）隔离 Tauri/Web | `platform/storage` 统一入口 |

### 7.2 最强约束速查

| 约束 | 内容 |
|------|------|
| **模块隔离** | 业务模块之间禁止相互依赖，仅通过 IntegrationEvent |
| **Service 合并** | 每个模块只有 1 个 Service 类（§21） |
| **统一响应** | 所有 API 返回 `ApiResult<T>`，HTTP 200 |
| **RPC 风格** | `[HttpPost] Create/Update/Delete/List`，`[HttpGet] Item`，禁止 PUT/DELETE/PATCH |
| **无路径参数** | URL 禁止 `{id}` 路径参数，统一用 query/body |
| **强类型** | 请求参数和响应字段使用 C# 强类型，枚举禁止用 string |
| **前后端字段一致** | 前端 key 与后端模型字段一致，禁止键名映射 |
| **API 客户端** | 前端必须通过 `apiFactory(Api).try()`，禁止 fetch/axios |
| **类型严格** | 前端禁止 `any`/`object`/`Record<string, any>` |
| **页面懒加载** | 所有页面 `React.lazy()` + `LazyComponent`，在 `menus.tsx` 注册 |
| **禁止 `--no-verify`** | git 操作绝对禁止跳过 hooks |
| **主键** | `long` + `IdGen.NewId()`（雪花 ID） |
| **数据库** | snake_case、3NF、外键约束、禁止逻辑删除 |
| **时区** | 客户端 HTTP Header `Timezone` 传递 IANA 时区 ID |
| **认证** | 自定义 AES Token，HTTP Header `token` |
| **项目隔离** | HTTP Header `ProjectId` → `DataFilterContext` → Repository 基类隔离 |
