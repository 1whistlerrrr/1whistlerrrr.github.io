# 进程 / 线程 / 分布式 — 一图看懂

## 单机单进程（你们现在的开发环境）

```
┌─── 一台机器（你的 MacBook）──────────────────────────────────┐
│                                                              │
│  ┌── 进程A：HostAPP（PID 62950）────────────────────────┐    │
│  │  端口：5050    内存：独立（256MB）                      │    │
│  │                                                       │    │
│  │  ┌─ 线程1 ─┐ ┌─ 线程2 ─┐ ┌─ 线程3 ─┐               │    │
│  │  │ 请求A    │ │ 请求B    │ │ 后台任务 │               │    │
│  │  │ 出库丙酮 │ │ 入库甲醇 │ │ 考勤生成 │               │    │
│  │  └─────────┘ └─────────┘ └─────────┘               │    │
│  │       │            │           │                      │    │
│  │       └─────┬──────┘           │                      │    │
│  │             ↓                  │                      │    │
│  │  共享：static 变量（_batchSeqIn, _batchSeqOut）       │    │
│  │  不共享：每个线程自己的栈（局部变量）                    │    │
│  └──────────────────────────────────────────────────────┘    │
│                                                              │
│  ┌── 进程B：PostgreSQL──────────────────────────────────┐    │
│  │  端口：5432    内存：独立（512MB）                      │    │
│  │  线程池：一堆 worker 线程处理 SQL 请求                   │    │
│  │  所有进程共享的真相来源（数据库里的数据）                  │    │
│  └──────────────────────────────────────────────────────┘    │
└──────────────────────────────────────────────────────────────┘

共享：DB 数据（跨进程）    static 变量（同进程内跨线程）
不共享：进程间内存完全隔离
```

```
│         │                    │  │         │                    │
│         │     ╳ 完全不共享  │  │         │                    │
│         │      内存相互隔离 │  │         │                    │
│         └────────┬─────────┘  │         └────────┬─────────┘  │
│                  │             │                  │            │
└──────────────────┼─────────────┘  ─────────────────┼────────────┘
                   │                                  │
                   └──────────┬───────────────────────┘
                              │ 网络
                              ↓
                   ┌─────────────────────┐
                   │  PostgreSQL         │  ← 唯一共享的
                   │  机器3（10.0.0.3）    │

```

## 真正的分布式数据库（TiDB 举例）

```
┌── 机器1：你的 App ──────────┐
│  HostAPP :5050               │
│  发 SQL：UPDATE stock ...    │
└──────────┬──────────────────┘
           │ 网络
           ↓
┌── 机器2：TiDB Server ──────────────────────────────────────┐
│  进程D：协调者                                               │
│  ┌──线程1────────────────────────────────────────────────┐ │
│  │ 收到 SQL → 解析 → 找到数据在 TiKV 节点3 → 发预提交请求    │ │
│  │ → 节点3 确认 → 发"全部提交" → 返回成功给 App              │ │
│  └───────────────────────────────────────────────────────┘ │
│  多个线程处理不同请求，共享进程内存                            │
└──────────┬──────────────────────────────────────────────────┘
           │ 网络
     ┌─────┼─────┐
     ↓     ↓     ↓
┌── 机器3  ──┐ ┌── 机器4 ──┐ ┌── 机器5 ──┐
│ TiKV 节点1  │ │ TiKV 节点2 │ │ TiKV 节点3 │
│ 进程E       │ │ 进程F      │ │ 进程G      │
│ 存 1/3 数据 │ │ 存 1/3 数据│ │ 存 1/3 数据│
└────────────┘ └───────────┘ └───────────┘

共享：无（TiDB Server 和 TiKV 是不同机器上的独立进程，通过网络通信）
不共享：所有内存
并发：TiDB 用 2PC（Percolator）协调跨节点事务 → 保证多节点原子性
```

```
现在                                        将来（如果上 TiDB / 多实例 PG）
──────────────────────────────────────     ──────────────────────────────────
单机 PostgreSQL                              TiDB / 分布式 PostgreSQL
┌────────────┐                              ┌──────┐ ┌──────┐ ┌──────┐
│ PG 进程     │                              │ 节点1 │ │ 节点2 │ │ 节点3 │
│ 所有表都在   │                              │       │ │       │ │       │
│ 这一个进程里 │                              └──┬───┘ └───┬──┘ └───┬──┘
└────────────┘                                 └─────┬──────┘       │
  BEGIN/COMMIT                                       │              │
  天然原子                                           │ 2PC 协调     │
  → UPDATE WHERE                                    │ 保证原子     │
    total >= qty                                    ↓              │
  就是行锁，                                        全部提交 or 全部回滚
  单机直接实现                                     ← 比单机多了网络开销
```

# 并发一致性❗️
如何保证线程之间的一致性（进程内部）、数据库内容的一致性（进程之间）



## 并发一致性 — 只用考虑数据在哪里一定要一致（最基本的数据库需要一致 ）

{% markmap %}
- 并发一致性
  - 进程内（线程之间）
    - ✅Interlocked → 整数计数比如`count++`：依赖 CPU 底层的 CAS（Compare-And-Swap）指令，无锁实现
    - ✅lock → 复杂多步操作：当操作包含多步（例如：先判断，再修改，最后写日志），无法用单条 CPU 指令完成时，必须用锁将整段代码包起来，强制线程排队
    - ConcurrentDictionary → 并发集合
  - 进程间（共享数据库）
    - ✅数据库本身原子性 `UPDATE WHERE qty >= @n` （数据库底层完成一步从读取-更改-写入一气合成）
      - 缺点：1.存在高并发秒杀数据库线程不够<redis> 2.分布式系统订单和库存不在同一个数据库<最终一致性、Seata>
    - 乐观锁 version → 先读后写
    - 悲观锁 SELECT FOR UPDATE → 强制串行
    - 唯一约束 → 防重复插入
  - 分布式（跨数据库）
    - 2PC 两阶段提交 → 强一致，TiDB 内置
    - TCC Try-Confirm-Cancel → 业务层补偿
    - 最终一致 + 对账 → 允许短暂不一致
  - 高并发秒杀
    - Redis Lua 预扣 → MQ 削峰 → DB 兜底
      - 1. 提前把商品库存加载到 Redis。
      - 2. 用户请求进来，执行 Redis Lua 脚本（保证扣减库存和判断的原子性）。
      - 3. 扣减成功后，立即返回给用户“排队中/抢购成功”，然后向 MQ（消息队列） 发送一条创建订单的消息（异步化，削峰填谷）。
      - 4. 后台消费者订阅 MQ，慢慢消化并写入 DB（数据库）。
{% endmarkmap %}
