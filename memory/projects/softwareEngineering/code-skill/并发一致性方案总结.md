# 并发一致性 — 完整方案总结

> 从单机内存到分布式数据库，从线程安全到事务隔离

---

## 零、先判断：我的并发问题属于哪一层？

```
你的代码里有一个变量/数据 → 会被"同时"修改 → 问自己三个问题：

Q1: 修改发生在同一个进程内吗？
    ├─ 是 → 看 §一（进程内并发）
    └─ 否 → Q2

Q2: 多个进程共享同一个数据库吗？
    ├─ 是 → 看 §二（数据库并发）
    └─ 否（没共享数据库，各自独立存储）→ 不需要处理，本来就各管各的

Q3: 多个进程 + 多个数据库 + 需要跨库事务？
    └─ 是 → 看 §三（分布式事务）
```

## 一、进程内并发（线程之间）— 单机内存

### 场景：多个线程同时改同一个变量

```
线程A ──→ _batchSeqIn = 15 ──→ 读 → 加1 → 写 16
线程B ──→ _batchSeqIn = 15 ──→ 读 → 加1 → 写 16
结果：两个人都拿到 16 ❌
```

### 方案对比

| 方案 | 怎么用 | 适用场景 | 你的项目用的地方 |
|------|-------|---------|:--:|
| **Interlocked** | `Interlocked.Increment(ref x)` | 简单数值加减 | `_batchSeqIn` / `_batchSeqOut` |
| **lock** | `lock(_obj) { ... }` | 复杂多步骤操作 | `_batchLock`（日期切换初始化） |
| **ConcurrentDictionary** | `dict.AddOrUpdate(...)` | 并发字典读写 | 批次号初版（后来删了） |

### 判断流程

```
需要保护的数据是什么类型？
├── 整数（计数、序号）→ Interlocked（最快，CPU 硬件指令）
├── 复杂对象（多字段修改）→ lock
├── 集合（增删改查）→ ConcurrentDictionary / ConcurrentQueue
└── 只读（初始化后不变）→ 不需要保护
```

## 二、进程间并发 — 同一数据库

### 场景：多个进程/实例操作同一行数据

```
进程A :5050        进程B :5051
   ↓                  ↓
   └──── 同一个 PostgreSQL ────┘
              ↓
    同一条 stock 记录，两个进程都要改
```

### 方案对比

| 方案 | QPS | 语句 | 怎么保证 | 适用场景 |
|------|:--:|------|---------|---------|
| **原子 UPDATE**（你用的） | ~2000 | `UPDATE SET qty = qty - @n WHERE qty >= @n` | PG 行锁 + WHERE 条件重求值 | 单条库存扣减 |
| **乐观锁**（version 列） | ~1000 | `UPDATE SET qty = qty - @n, version = version + 1 WHERE version = @old` | 冲突检测 + 重试 | 需要先读后写的场景 |
| **悲观锁** | ~300 | `SELECT ... FOR UPDATE` | 显式锁住行直到事务结束 | 资金类强制串行 |
| **唯一约束** | — | `UNIQUE (project_id, batch_no)` | DB 拒绝重复写入 | 防重复插入 |

### 判断流程

```
操作类型？
├── 单步扣减（减库存）→ 原子 UPDATE WHERE qty >= @n ⭐（最简）
├── 先查后改（读取 → 判断 → 修改）→ 乐观锁 version 列
├── 必须串行（转账）→ 悲观锁 SELECT FOR UPDATE
├── 防重复（批次号唯一）→ 唯一约束
└── 需要和别的表操作同一事务 → 原子 UPDATE + 事务包裹
```

### 你的库存扣减为什么选原子 UPDATE

```sql
-- 这一条语句就是你的全部并发方案
UPDATE material_inventory_ledger
SET total_quantity = total_quantity - @qty
WHERE project_id = @p AND material_id = @m AND total_quantity >= @qty;

-- affected == 0 → 库存不足
-- 不需要先 SELECT
-- 不需要 version 列
-- 不需要 FOR UPDATE
-- PG 的行锁 + 重新求值 WHERE 自动处理并发
```

## 三、分布式事务 — 跨多个数据库

### 场景：一个操作涉及两个独立数据库

```
UPDATE 库A.表X SET qty = qty - 10;   → 节点A
UPDATE 库B.表Y SET qty = qty + 10;   → 节点B
-- 节点A 写成功了，节点B 网络断了 → 数据不一致！
```

### 方案对比

| 方案 | 性能 | 复杂度 | 适用 |
|------|:--:|:--:|------|
| **2PC**（两阶段提交） | 低 | 高 | 强一致性必须场景，TiDB 内置 |
| **TCC**（Try-Confirm-Cancel） | 中 | 极高 | 业务层补偿，阿里 Seata |
| **最终一致 + 对账** | 高 | 低 | 允许短暂不一致，定时修数据 |

### 判断流程

```
必须跨库事务？
├── 是 + 数据量巨大 + 可接受短暂不一致 → 最终一致 + 定时对账
├── 是 + 必须实时一致 → 2PC / TCC
└── 否 → 你不需要这层，看 §一 §二就够了
```

**你们项目当前不需要**：所有表在同一个 PostgreSQL 里，单机事务天然原子。

## 四、消息队列异步 — 削峰场景

### 场景：秒杀，1 秒 10 万个请求

```
前端请求 → Redis Lua 预扣（极快，QPS ~100000）
              ↓ 成功
           MQ 消息队列（削峰，排队慢慢处理）
              ↓
           DB 最终落库（CHECK 约束兜底）
```

### 方案

| 层 | 用什么 | 作用 |
|----|-------|------|
| **第一层** | Redis Lua 脚本 | 原子预扣，抗量 |
| **第二层** | MQ（RabbitMQ/Kafka） | 削峰填谷，异步化 |
| **第三层** | DB 原子 UPDATE + CHECK | 最终落库，防超卖兜底 |
| **对账** | 定时任务 | Redis 存量 vs DB 存量对比，修复差异 |

## 五、全场景判断决策树

```
你的数据被并发修改了 → 判断在哪一层：

1. 只在进程内的变量（static、字段、集合）
   → Interlocked / lock / ConcurrentDictionary

2. 在数据库里，单机 PostgreSQL
   → 原子 UPDATE WHERE qty >= @n（首选）
   → 乐观锁 version（需先读后写）
   → 悲观锁 SELECT FOR UPDATE（强串行）

3. 在 Redis 里
   → Lua 脚本（Redis 单线程执行，天生原子）

4. 跨多个数据库 / 跨 Redis + DB
   → 2PC / TCC / 最终一致 + 对账

5. 高并发秒杀（QPS > 10000）
   → Redis Lua 预扣 → MQ 削峰 → DB 兜底
```
