## 一、一句话概括你做了什么

> 我为智慧工地平台设计并实现了一套材料物资的出入库管理系统，核心是通过 **单条 SQL 原子 UPDATE** 解决多仓库并发出库不超扣的问题，同时用**物化库存台账**替代实时聚合保证了查询性能，用**归档表**代替逻辑删除满足了审计追溯要求。

## 二、面试官会问的问题（按深度递进）

### Q1: 先介绍一下整体设计？

**回答要点**：8 张表，5 个子域，一个 Service

```
数据库：material_dict（字典） → materials（材料） → warehouses（仓库）
         ↓                          ↓
  material_inventory_ledger   warehouse_material_stock
  （按材料+项目的库存总量）      （按仓库+材料+项目的存放量）
         ↑                          ↑
  inout_orders（出入库流水主表） → inout_order_details（明细子表）
         ↓
  inout_orders_archive（软归档，删除时 JSON 快照迁入）
```

**关键设计决策**：库存不实时 SUM 明细，而是**物化存储**两个汇总表。出入库保存时在事务内增量更新，查询时直接读物化值，P95 < 500ms。

---

### Q2: 出库并发不超扣怎么做的？这是核心问题

**当前实现**（直接拿代码说）：

```sql
-- 原子扣减，单条语句，不先读再写
UPDATE material_inventory_ledger
SET total_quantity = total_quantity - @qty,
    accrued_out_quantity = accrued_out_quantity + @qty,
    updated_at = now()
WHERE project_id = @p
  AND material_id = @m
  AND total_quantity >= @qty;

-- 仓库维度同理
UPDATE warehouse_material_stock
SET stock_quantity = stock_quantity - @qty
WHERE project_id = @p
  AND warehouse_id = @w
  AND material_id = @m
  AND stock_quantity >= @qty;
```

**为什么能防超卖**：

```
时间线：
事务A（出库8）：UPDATE ... WHERE total>=8 → total=10够 → 扣减成功，行锁持有
事务B（出库8）：UPDATE ... WHERE total>=8 → 等A释放行锁...
事务A提交 → 释放锁，total变为2
事务B拿到锁 → 重新求值 WHERE total>=8 → 2>=8 为假 → affected=0 → 库存不足
```

**关键点**：PostgreSQL 默认 READ COMMITTED 隔离级别下行级写锁 + 重新求值 WHERE 条件。B 事务看到的是 A 提交后的最新值，不是自己读的快照。

**为什么不用其他方案**：

| 方案 | 为什么没选 | 
|------|-----------|
| SELECT FOR UPDATE 悲观锁 | 多一次往返，锁持有时间更长，吞吐更低 |
| 乐观锁 version 列 | 需要先 SELECT 再 UPDATE，两次往返，冲突要重试循环 |
| Redis 分布式锁 | 项目没有 Redis，引入新中间件成本高 |
| 应用层 `lock` / 内存锁 | 单实例无效，将来多实例部署怎么办 |

**数据库兜底**：
```sql
CONSTRAINT chk_mil_total_nonneg CHECK (total_quantity >= 0)
```
即使代码逻辑有 bug，数据库也不会写入负数库存。

### Q3: 那如果多个仓库同时出库同一种材料呢？

两种库存维度独立控制：

```
material_inventory_ledger（全局）：total = 100（丙酮总库存）
warehouse_material_stock（按仓库）：
  1号库 stock = 30
  2号库 stock = 70

出库场景：
  从1号库出20 → 检查1号库 stock>=20 ✓ → 扣1号库，total 100→80 ✓
  从1号库出40 → 虽然 total=80>=40，但1号库 stock=10<40 → failed ✗
```

两个 UPDATE 在同一事务内，同时满足才提交。不会出现"全局够但仓库不够"被放行的情况。

### Q4: 出入库流水为什么编辑和删除这么复杂？

因为**库存是物化的**，不是实时算的。

**编辑**（回滚旧 + 应用新）：
```
事务内：
  1. 查旧流水（原来是入库100kg丙酮到1号库）
  2. C-001 前置校验：当前1号库 stock=70 < 100？
     → 是！因为中间出过30，回滚100会让stock变负
     → 回滚事务，返回友好错误「该入库已被后续出库消耗，无法编辑」
     → 不依赖 CHECK 约束抛 500
  3. 校验通过 → 回滚旧：total-=100, stock-=100
  4. 删除旧明细 → 应用新（80kg）：total+=80, stock+=80
  5. Commit
```

**删除**（软归档 + 回滚）：
```
事务内：
  1. C-001 前置校验（同上）
  2. 回滚库存（入库单扣回，出库单加回）
  3. 序列化主子表为 JSON → INSERT inout_orders_archive
  4. DELETE inout_orders（明细 FK CASCADE 自动删）
  5. Commit
```

**为什么不用 `is_deleted` 字段？**

数据库建模规范**禁止逻辑删除字段**，要求"真实 DELETE 或归档表"。归档表存 JSON 快照，满足 NFR-008「不可物理丢失」，且活动表查询无需 `WHERE is_deleted=false`。

### Q5: 为什么入库 UPSERT、出库 UPDATE 要用原生 SQL 而不是 EF Core？

因为 EF Core 做不到一条语句完成这些操作：

```sql
-- 入库 UPSERT：EF Core 8 没有原生 UPSERT API
INSERT INTO material_inventory_ledger (...)
VALUES (...)
ON CONFLICT (project_id, material_id)
DO UPDATE SET total_quantity = ledger.total_quantity + @qty, ...

-- 出库原子扣减：ExecuteUpdateAsync 支持 WHERE，但需要先查再用返回的 affected 判断
-- 而且要和仓库维度扣减在同一事务 → 用 Npgsql 原生事务最直接
```

**为什么合规**：参照项目范本 SafetyTechManagement，Service 通过 `IDbInfrastructure`（Repository 层基础设施，只提供连接字符串 + 事务）执行原生 SQL，不直接注入 `AppDbContext`。这是项目既有模式。

### Q6: 批量出库同一张流水里多行同一种材料怎么处理？

**合并后一次扣减**，不逐行扣：

```csharp
// 明细：丙酮 30 + 丙酮 20 → 合并为丙酮 50，一次 UPDATE
var grouped = req.Details.GroupBy(d => d.MaterialId)
    .Select(g => new { MaterialId = g.Key, TotalQty = g.Sum(d => d.Quantity) });
```

为什么：如果逐行扣，第一行扣 30 后 total 变了，第二行扣 20 可能因并发看到不准确的值。合并后一次扣 50，减少和数据库的交互次数，也避免中间态误判。

## 三、将来可能遇到的问题 & 演进方向

### 问题1：当前方案在多实例部署下的局限性

**当前**：单机 DB 乐观锁，所有操作走 PostgreSQL。

**如果上多实例**：DB 乐观锁仍然有效（行锁是 DB 层的，跨实例），但 QPS 受限于 DB 连接数和 IO：

```
实例1  ↓ 实例2  ↓ 实例3
   \    |    /
    PostgreSQL（瓶颈点）
```

**演进方向**：加 Redis 做热库存缓存 + Lua 原子扣减，DB 定期同步。

### 问题2：当前方案在秒杀 / 高并发场景下的性能瓶颈

**当前 QPS**：DB 乐观锁 ≈ 500-2000，智慧工地场景够用。

**如果上电商级高并发**：
```
第一层 Redis Lua 预扣（抗量，QPS ~100000）
  → 成功 → 第二层 MQ 异步削峰
    → 第三层 DB 最终落库 + CHECK 约束兜底
```

### 问题3：物化库存和明细数据不一致怎么处理

虽然同一事务保证强一致，但极端情况（如 DBA 手动改数据）可能导致台账和明细对不上。

**排查 SQL**：
```sql
-- 台账总量 vs 明细 SUM 应该相等
SELECT m.id, m.material_name,
  l.total_quantity AS ledger_total,
  COALESCE(SUM(CASE WHEN o.inout_type=1 THEN d.quantity ELSE -d.quantity END), 0) AS detail_sum
FROM materials m
JOIN material_inventory_ledger l ON l.material_id = m.id
LEFT JOIN inout_order_details d ON d.material_id = m.id
LEFT JOIN inout_orders o ON o.id = d.order_id
GROUP BY m.id, l.total_quantity
HAVING l.total_quantity != COALESCE(SUM(...), 0);
```

### 问题4：软归档数据越来越多怎么办

`inout_orders_archive` 存 JSON 快照，理论上无限增长。

**当前**：工地场景删除操作少，短期无问题。
**将来**：加定时任务，超过 N 天的归档数据迁移到冷存储（对象存储 / 数据仓库），活动表只保留近期的。

### 问题5：为什么没有用 Redis？什么时候该加？

**当前不需要的理由**：
- 工地管理场景 QPS < 100，DB 乐观锁绰绰有余
- Redis 是额外运维负担（哨兵/集群、主从切换丢了怎么办）
- 库存扣减必须和流水写入同一事务，Redis 不能跨 Redis + DB 做事务

**该加的信号**：
- DB CPU 持续 > 70%，慢查询里库存 UPDATE 占比高
- 业务方提需求：要做限时抢购 / 秒杀

**加的时候怎么加**：Redis Lua 做第一层预扣，MQ 异步落 DB，DB CHECK 约束兜底。不是"替换"方案，是"叠加"方案。

## 四、面试回答模板

**面试官：** "你们这个出入库系统，并发怎么处理的？"

> 我们用 PostgreSQL 的单条 UPDATE ... WHERE total_quantity >= @qty 做原子扣减。A 事务扣减时对行加写锁，B 事务会等 A 提交后基于最新值重新求值 WHERE，不满足就直接返回库存不足。数据库还加了 CHECK(total_quantity >= 0) 约束兜底。这个方案在目前工地场景 QPS 不高的情况下完全够用，不需要引入 Redis。

**面试官：** "那多人同时操作同一个仓库的同一种材料呢？"

> 全局库存和仓库维度库存是分开控制的。即使全局总量够，如果某个具体仓库的库存不够，出库也会被阻止。两个 UPDATE 在同一事务内，不满足就回滚。

**面试官：** "如果以后要上多实例呢？"

> 行锁是 DB 层的，多实例也不影响。真正的瓶颈是 DB 连接数，量大了就加 Redis Lua 做第一层预扣，异步落 DB，相当于在 DB 乐观锁外面包一层缓存。

**面试官：** "你遇到的最棘手的 bug 是什么？"

> 编辑入库单的时候，如果那个入库已经被后续出库消耗了一部分，回滚库存会让 stock 变成负数，数据库 CHECK 约束会抛 500。我在回滚前加了前置校验，读当前 stock，如果不够就返回友好错误"已被后续出库消耗，无法编辑"而不是让用户看到 500。
