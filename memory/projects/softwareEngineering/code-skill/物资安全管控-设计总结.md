## 二、架构决策（5 条 ADR）

| 编号 | 决策 | 核心思想 |
|------|------|---------|
| ADR-001 | 单语句原子 `UPDATE … WHERE stock >= @qty` | PostgreSQL READ COMMITTED + 行锁保证并发不超扣 |
| ADR-002 | 库存双维度物化存储 | 台账表(全局) + 存放量表(按仓库)，写入增量更新，读取 O(1) |
| ADR-003 | 软归档 = 归档表 + 库存回滚 | JSON 快照写入独立表，活动表物理删除，满足 NFR-008 不可物理丢失 |
| ADR-004 | 三类字典合并单表 | `material_dict` + `dict_type` 区分，避免三张同构表 |
| ADR-005 | 增量 DDL SQL 脚本 | 不用 EF Migration，追加 `upgrade.sql`，与全仓库一致 |

## 三、子域划分

```
MaterialSafety
├── MaterialDict   — 字典 CRUD（材料分类/单位/规格），改名同步材料快照(H-002)
├── Material       — 材料主数据 CRUD，删除前被引用检测
├── Warehouse      — 仓库/存放点 CRUD + GIS 坐标 + 视频设备关联
├── Inventory      — 库存台账只读查询 + 四区域详情聚合
└── InoutOrder     — 出入库流水主子表，保存即生效，事务内增量更新库存
```

### 依赖关系

- **InoutOrder 写** → Inventory（台账/存放量事务内 UPSERT/原子扣减）
- **InoutOrder 校验引用** → Material + Warehouse
- **Material 引用** → MaterialDict（FK RESTRICT）
- **Inventory 读** → InoutOrder（详情弹窗出入库记录分页）+ Warehouse（存放详情）

## 四、核心业务流程

### 出库不超扣（ADR-001）

```
UPDATE material_inventory_ledger
SET total_quantity = total_quantity - @qty,
    accrued_out_quantity = accrued_out_quantity + @qty
WHERE project_id = @p AND material_id = @m AND total_quantity >= @qty
→ affected == 0 → 库存不足 → 回滚
```

### 编辑流水（回滚旧 + 应用新）

```
事务内：
  1. 加载旧流水 + 旧明细
  2. C-001 前置校验（入库回滚前检查 total >= oldQty）
  3. 回滚旧影响（入库扣减、出库加回）
  4. 删除旧明细
  5. 应用新影响（同 Create）
  6. 更新主表
  7. Commit
```

### 删除流水（软归档 + 回滚）

```
事务内：
  1. C-001 前置校验
  2. 回滚库存
  3. 写归档表（JSON 快照）
  4. 物理删除主表（明细 CASCADE）
  5. Commit
```

## 六、实现过程中的关键经验

1. **FilterQL vs query 参数**：FilterQL 用于模糊搜索（CONTAINS），query 参数用于精确值筛选（枚举/ID）——设计 spec 中有明确定义，不要混用
2. **枚举序列化**：全局 `JsonStringEnumConverter(CamelCase)` 自动处理，数字存 DB，camelCase 字符串走 API
3. **匿名类型 + PagingFetch**：JOIN 查询产生的匿名类型无法被 FilterQL 解析，需要手动提取 FilterQL 条件后置空再传
4. **Native SQL 事务**：原子扣减需要单语句 `WHERE` 条件，EF 的 `ExecuteUpdateAsync` 无法满足，需用 Npgsql 原生 SQL + 同一连接事务
5. **C-001 前置校验**：编辑/删除入库单前必须检查 `total >= oldQty`，不能依赖 CHECK 约束兜底返回 500
6. **FK 同名**：`HasConstraintName` 与 DDL 约束名一致，避免 EF 二次生成默认名
7. **HostAPP.csproj + ModulesLoader**：新增模块必须同时加项目引用 + `_modules` 列表注册

### Npgsql 原生 SQL 事务 ⭐️
```csharp
// 为什么：原子扣减需要 UPDATE ... WHERE total >= @qty 单语句语义
// EF ExecuteUpdateAsync 无法满足 → 用 Npgsql 原生 SQL
using var conn = new NpgsqlConnection(connStr);
await conn.OpenAsync(ct);
using var tx = await conn.BeginTransactionAsync(ct);
// 所有操作（UPSERT/UPDATE/INSERT）走同一连接+事务
await tx.CommitAsync(ct);
```

### `lock` 内不能 `await`
```csharp
// ❌ 编译错误 CS1996
lock (_lock) { await SomethingAsync(); }

// ✅ 正确：把 async 操作移到 lock 外，lock 只做状态修改
var data = await QueryDbAsync();
lock (_lock) { UpdateState(data); }
```

## 新增接口：GenerateBatchNo — 批次号生成

### 需求迭代过程
1. **初版**：`CL` + YYMMDD + 4位流水，查 DB MAX + 1
2. **争议**：加了 ConcurrentDictionary 内存计数器+DB 双保险 → 过度设计，被要求回退
3. **最终版**：Interlocked.Increment 线程安全递增 + 日期切换时从 DB 初始化 + 出入库区分前缀

### 出入库区分前缀
- 入库：`CLRK2607150001`（RK = 入库）
- 出库：`CLCK2607150001`（CK = 出库）
- 两个独立计数器（`_batchSeqIn` / `_batchSeqOut`），互不干扰
- 日期取自请求 `Timezone` Header，支持跨时区

### 关键代码模式
```csharp
// 线程安全递增（无锁竞争）
int nextSeq = Interlocked.Increment(ref _batchSeqIn);

// 日期切换时 double-check + lock 初始化
if (_batchDate != datePart)
{
    lock (_batchLock)
    {
        if (_batchDate != datePart)  // double-check
        {
            // 从 DB 取当天最大值初始化两个计数器
            // ...
            _batchDate = datePart;
        }
    }
}
```

## 筛选方式 — FilterQL vs query 参数

### 教训：擅自修改导致与 spec 不一致
- **错误做法**：把 category/dictType 等精确筛选从 query 参数改为 FilterQL
- **问题**：设计 spec §5 明确规定了每个接口的参数位置
- **回退**：FilterQL 只用于 `CONTAINS` 模糊搜索；`=` 精确筛选用 query 参数
- **教训**：改动代码前先读 spec，不要自行发挥

### 当前规范
| 筛选类型 | 传参方式 | 示例 |
|---------|---------|------|
| 模糊搜索（名称/批次号/地址） | FilterQL（body） | `"materialName CONTAINS '丙'"` |
| 精确值（分类/单位/规格/类型） | query 参数 | `?category=易燃液体&unit=kg` |
| 跨表 ID（责任单位/材料联动） | query 参数 | `?responsibleOrgId=3001` |
| 时间范围 | query 参数 | `?startTime=...&endTime=...` |

## `allowNoPaging` 参数详解

```csharp
// PagedQuery.cs 源码逻辑
if (option.NoPaging && allowNoPaging == false)
    throw new NotSupportedException("NoPaging is not allowed.");

// allowNoPaging 是服务端的安全开关
// false → 禁止客户端 noPaging:true，抛异常
// true  → 允许全量返回
```

| 场景 | 设置 | 原因 |
|------|:--:|------|
| MaterialDict（≤100条） | `true` | 字典数据量小，全量返回没问题 |
| InoutOrder（可能上万条） | `false` | 防止客户端请求百万数据打崩服务 |

## 服务启动故障排查

### 遇到的错误 & 根因
| 错误 | 表面现象 | 根因 |
|------|---------|------|
| `OperationCanceledException` | VoiceprintService 崩溃 | `PeriodicTimer` 取消没 catch，MassTransit DB 连接失败触发 |
| `ObjectDisposedException` | 考勤服务报 IServiceProvider 已释放 | Host 正在关闭，服务间连锁反应 |
| `Address already in use` | Kestrel 绑定端口失败 | 旧进程未退出就启动新进程 |
| `AddressInUseException` | PostgreSQL `remaining connection slots are reserved` | 反复重启耗尽连接池 |

### 正确流程
```bash
# 1. 彻底杀掉旧进程
lsof -ti :5050 | xargs kill -9
# 2. 等 5 秒（MassTransit SQL Transport 需要清理连接）
sleep 5
# 3. 重新启动
dotnet run --project HostAPP --urls http://localhost:5050
```
