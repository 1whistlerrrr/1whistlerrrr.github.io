## 核心设计模式

### 1. 表分类体系

| 类型 | 用途 | 特征 |
|------|------|------|
| 字典表 | 可 CRUD 的枚举值 | `dict_type` 分组，项目隔离 |
| 主数据表 | 业务对象基础信息 | 被其他表引用，有唯一约束 |
| 核心实体表 | 业务事件记录 | 含 FK 到主数据，有冗余快照字段 |
| 汇总台账表 | 物化聚合值 | 由事件表事务内增量更新，只读查询 |
| 归档表 | 删除留痕 | JSON 快照，无 FK，独立于活动表 |

### 2. API 8 段式模板

```
#### API-xxx: Controller/Method

**URL**: METHOD /api/Module/Controller/Method

**请求参数**: 表格（参数名/类型/必填/位置/说明）

**业务逻辑**: 编号步骤

**响应 data 字段**: 表格（字段名/类型/必返/说明）

**请求示例**: 完整 JSON

**成功响应示例**: 完整 JSON

**错误响应示例**: 至少 3 个场景的 JSON

**错误码**: Code + Message + 触发条件
```

### 3. ADR 四段式

```
背景：当前问题和技术约束
决策：选择什么方案，怎么实现
理由：为什么选这个 / 为什么不选备选
后果：正面影响 + 负面影响及注意事项
```

### 4. 约束编号化

每个设计决策分配编号（C-xxx / ADR-xxx），可追溯。在变更记录中引用。

## 关键约束（来自项目架构规范）

1. 仅 GET/POST，RPC 风格，PascalCase 方法名
2. `ApiResult<T>` 统一响应，复用现有 ErrorCode
3. 所有实体带 `long ProjectId`，仓储继承 `DirectKeyRepositoryBase`
4. 每模块单一 Service 类，`#region` 分子域
5. 无 EF Migration，DDL 追加 `upgrade.sql`
6. 无逻辑删除（`is_deleted`），用归档表
7. FK 与 DDL 同名（`HasConstraintName`）
8. 枚举序列化为 camelCase 字符串（全局 `JsonStringEnumConverter`）
9. FilterQL 用于模糊搜索（CONTAINS），query 参数用于精确筛选
