# 个人工作流总结 — Harness 迭代反馈机制

> 每次对话完成后，提取"这套 harness 还缺什么"，驱动 harness 持续迭代
> 当前版本：基于物资安全管控模块对话（2026-07-14~16，21 轮对话）

---

## 一、核心理念

```
每次对话 = 一次 harness 压力测试

对话中：
  ✅ 顺利完成的 → harness 已经做对了，保持
  ❌ 反复纠正的 → harness 缺了什么，需要补充
  🔁 每次都要重复说的 → 应该固化到规范/模板/脚本里

对话后：
  提取 → 分类 → 写入本总结 → 驱动 harness 更新
```

---

## 二、本次对话暴露的 Harness 缺口

### 缺口 1：Agent 不知道"什么能做 / 什么不能做"

| 暴露方式 | 具体场景 |
|---------|---------|
| 反复纠正 | 擅自把 query 参数改为 FilterQL，后来才发现 spec 有明确规定 |
| 踩坑后补 | 枚举序列化是 camelCase 字符串，文档写了 number |
| 编译报错 | `lock` 内 `await`（C# 限制） |

**根因**：没有一份"项目铁律"文档让 Agent 在编码前必须读完。

**建议**：新建 `规范/项目铁律.md`，包含 11 条。所有后端编码 Skill 第一行引用。

### 缺口 2：Agent 不知道"照着谁写"

| 暴露方式 | 具体场景 |
|---------|---------|
| 每次 Prompt 都要写 | 14 个范本文件路径，每个 Phase 都要重复列一遍 |
| 容易漏 | Phase1/2/3 的 Prompt 里路径描述不统一 |

**根因**：没有注册表。Agent 不知道哪个文件是权威范本。

**建议**：新建 `规范/范本文件映射表.md`，按"你要创建什么 → 照着哪个文件写"的格式。

### 缺口 3：漏改现有文件导致编译失败

| 暴露方式 | 具体场景 |
|---------|---------|
| 编译报错 | Phase2 忘了在 HostAPP.csproj 加 ProjectReference → CS0246 |
| 事后补 | sln 注册是用户提醒后才做的 |

**根因**：Agent 专注于创建新文件，不知道还有"必须修改的现有文件"。

**建议**：新建 `规范/新建模块-必须修改文件清单.md`（6 处）。Phase2 Prompt 必须包含此清单。

### 缺口 4：API 文档生成没有标准流程

| 暴露方式 | 具体场景 |
|---------|---------|
| 格式不一致 | 第一次生成太啰嗦，用户要求精简 |
| 类型写错 | 枚举类型写了 number，实际是 string |
| 缺少 Header 说明 | 用户要求补充 header 配置后才加 |

**根因**：API 文档是让 Agent 自由发挥的，没有模板约束。

**建议**：建 `templates/API接口文档模板.md`（8 段式标准格式）+ 写规范"生成后必须读 Json.cs 确认枚举类型"。

### 缺口 5：没有编码后的自动验证

| 暴露方式 | 具体场景 |
|---------|---------|
| 人工检查 | spec 一致性对比是用户要求后才做的 |
| 逐条确认 | 6 处必须修改是否都改了，靠人回忆 |

**根因**：编码完成后没有自动化的检查步骤。

**建议**：过程路由新增"阶段 5：编码后自动验证"，包含 dotnet build / csproj 引用检查 / ModulesLoader 检查 / sln 检查 / spec-代码对比。

### 缺口 6：反复重启导致 DB 连接池耗尽，无诊断文档

| 暴露方式 | 具体场景 |
|---------|---------|
| 用户困惑 | HostAPP 报错不知道原因，贴了一大堆 stack trace |
| 时间浪费 | 反复 kill 进程重试，没意识到需要等 MassTransit 释放连接 |

**根因**：没有"常见启动故障速查"文档。

**建议**：补充到 `规范/` 或 harness 的 README。

## 三、反复犯错的模式（应固化为 Harness 规则）

| 错误 | 出现次数 | 已在 Harness 中预防？ |

## 五、对话后的标准动作（每次对话结束执行）

```
每次模块开发对话完成后，编排者应：

1. 提取：
   □ 本次新增了哪些代码/文档 → 记录到进度文档
   □ 哪些地方反复纠正/踩坑 → 记录到"反复犯错模式"表
   □ 哪些 Prompt 每次都要重复写 → 记录到"待固化"列表

2. 分类：
   □ P0（不补就反复踩坑）→ 本周内建规范/脚本
   □ P1（可自动化提速）→ 下个迭代加入
   □ P2（方法论缺失）→ 下次类似对话后提炼

3. 更新：
   □ 本总结文档的"Harness 迭代记录"表
   □ 进度文档中的"已完成/待完成"
   □ 如有新规范/模板/脚本 → 登记到 harness-manifest.json
```

## 六、长期目标

```
3 次模块开发后 → 铁律 + 范本表 + 必须修改清单 固化完成
               → 新模块 Agent 第一次就能编译通过

5 次模块开发后 → 自动化检查脚本就位
               → 编码完成 = dotnet build + spec 一致性 + 6 处修改 自动验证

10 次模块开发后 → 新模块 SOP 可填空化
                → 人只需：给模块名 + 设计 spec 路径
                → Agent 自动：生成代码 + 注册 + 编译 + 跑测试 + 出报告
```


ai：重复的部分 提供方案 
人类：需要分层、细化的内容


harness （大模型靠梯度进行参数更新，agent可以靠skill提示词的更新）
