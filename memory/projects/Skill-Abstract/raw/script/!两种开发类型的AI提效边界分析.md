# 两种开发类型的 AI 提效边界分析

> 对比来源：
> - Type A：考勤/工时模块修复（跨模块逻辑修复，30 会话 427 轮，~70h）
> - Type B：物资安全管控模块（新模块代码生成，2 会话 93 轮，~10-15h 实际投入）

---

## 一、两种类型的本质区别

| 维度 | Type A：跨模块逻辑修复 | Type B：新模块代码生成 |
|------|------|------|
| **类比** | 修一栋已经住了人的楼的水管——你不知道墙里面管子怎么走的，关了总阀会影响到哪几户 | 在一块空地上按图纸盖新房——图纸画好了，材料规格定了，照着盖就行 |
| **输入质量** | 低——问题描述通常是 symptom（"返回了段之前的值"），不是根因 | 高——有详细设计 spec，API/数据库/字段类型全部明确 |
| **代码起点** | 既有代码（可能有历史包袱、不一致的设计、隐蔽的耦合） | 零（greenfield） |
| **影响范围** | 不确定——改 WorkerOnline 的算法，Attendance 模块也受影响 | 确定——新模块只在 `MaterialSafety.API/` 下，边界清晰 |
| **验证方式** | 回归验证——怕把原有功能 break 了 | 正向验证——按 spec 逐条确认是否实现正确 |
| **AI 的认知负担** | 极高——需要理解既有代码的意图、耦合关系、执行时的行为 | 低——spec 描述清楚了要什么，按模板生成即可 |
| **人的认知负担** | 高——需要判断"这个 symptom 背后的根因是什么"，需要做架构决策 | 中——需要确认 spec 是否正确，判断产出是否符合预期 |
| **每次修改的风险** | 高——可能引入回归 | 低——新模块不影响其他模块 |

## 二、Type A（跨模块修复）：AI 提效阶梯

### 当前状态：~70h

```
发现问题 → 贴日志给 AI → AI 分析 → 人看不懂分析 → 再问 "这是什么意思"
    → AI 给方案 → 人确认 → AI 改代码 → 编译 → 手动 curl 测试
    → 发现新问题（改了 A 但 B 也受影响）→ 循环
```

### 第一级：流程优化（可达 ~40h，省 43%）

**不需要改 harness，只需要改变人的行为：**

| 改变 | 具体做法 | 省时来源 |
|------|---------|---------|
| **根因分析前置** | 发现 symptom 后第一轮不要求改代码，只要求 AI 列出所有可能原因并逐一分析 | 避免"改错了方向再回头"，WorkerOnline 69 轮 → 预计 20 轮 |
| **影响范围分析** | 每次修改前强制 AI 搜索整个代码库中相似的逻辑 | 避免"改了 A 漏了 B"，省掉后面的补丁轮次 |
| **学习与修复分离** | 遇到不懂的概念（Singleton/Scoped、时区）→ 记下来 → 开独立学习会话 | 不占用修复会话的认知带宽 |

**这一级不需要改 harness。只需要人改变交互习惯。**

### 第二级：工具化（可达 ~25h，省 64%）

**需要 harness 增加轻量工具：**

| 工具 | 形式 | 触发时机 | 效果 |
|------|------|---------|------|
| **跨模块耦合搜索** | Skill（非 Agent） | 修改任何 Service 方法前，自动搜索所有调用该方法的位置 + 所有实现相同接口/模式的其他类 | AI 在动手改代码前就看到"这个改动会影响 X 个文件"，不再遗漏 |
| **一键回归测试生成** | Skill | 在发现 bug 的第一轮对话中，AI 自动生成覆盖所有相关接口的测试脚本 | 每次修改后 30 秒跑完回归，不用手动 curl |
| **环境健康快检** | Script（bash） | `init.sh` 增加 `--quick-check` 模式（编译主项目 + DI 冲突扫描 + DB 连通性） | 修改前知道基线状态，修改后知道是否引入新问题 |

**这一级的关键：工具应该是轻量的 Skill 或 Script，不需要独立 Agent。**

### 第三级：Agent 智能化（可达 ~18h，省 74%）

**需要 harness 改造 Agent：**

| 改造 | 内容 |
|------|------|
| **coder agent 内置影响范围分析** | 在 Agent prompt 中加入强制步骤："修改任何已有方法前，先 Grep 搜索所有调用点 + 所有同名/同模式的方法，输出影响范围清单" |
| **类型安全检查** | Agent 中增加 Npgsql 类型映射规则：LINQ 投影中禁止匿名类型属性与 DB 列类型不匹配 |
| **DI 生命周期检查** | Agent 中增加规则：`AddSingleton<T>` 的 T 的构造函数依赖必须全部是 Singleton |

### 天花板：为什么省不到 80% 以上

| 不可替代的部分 | 占比 | 原因 |
|------|:--:|------|
| **根因判断** | ~15% | "返回了段之前的值" → 人需要判断这是时区问题还是算法问题，这个判断依赖业务理解 |
| **架构决策** | ~10% | Singleton 还是 Scoped？回溯要不要去掉？这些是设计决策，AI 只能给建议 |
| **业务规则确认** | ~8% | "零点按 timezone 隔断" 是业务需求，不是技术问题 |

**Type A 的 AI 极限：省 ~70-75%，人仍然需要投入 ~18-20h 做判断和决策。**

## 三、Type B（新模块代码生成）：AI 提效阶梯

### 当前状态：~10-15h

```
详细设计 spec → Phase 1 Agent (枚举+实体+DDL) → Phase 2 Agent (仓储+基础设施)
    → Phase 3 Agent (Service+Controller) → 编译修复 → API 文档生成
    → curl 集成测试 → 单元测试 → 设计一致性验证
```

当前 AI 已经承担了 ~80% 的编码工作量。剩下的 ~20% 主要是修复 AI 的遗漏和错误。

### 第一级：消除低级遗漏（可达 ~5-7h，省 50%）

| 遗漏类型 | 当前频率 | 修复方式 |
|---------|:--:|------|
| 漏 `.csproj` ProjectReference | 每次新模块都会漏 | **coder agent 强制 check**：编码后自检清单（.csproj/Module/DDL/sln 全链路） |
| 枚举序列化假设错误 | 每次都会假设错（number vs camelCase string） | **编码前强读** `AccessCore/Shared/Json.cs`，提取全局序列化配置 |
| `PagingFetch` 的 `allowNoPaging` 默认值 | 逐个接口改 | **编码前强读** 至少 2 个已有 Controller，复制分页参数配置 |
| 单元测试 26% skip 率 | 每次都会这样 | **测试分类前置**：写测试前先输出分类表，纯逻辑 → 单元测试，IQueryable → 集成测试 |

**这一级的关键：agent prompt 中做硬性检查，不需要新增 Agent。**

### 第二级：全自动流水线（可达 ~2-3h，省 80%）

```
详细设计 spec
    ↓
【自动】env-check（编译基线 + DI 状态 + DB 连通性）
    ↓
【自动】backend-coder（编码 + 自检 build + 输出完整性报告）
    ↓
【自动】integrity-check（.csproj/Module/DDL/sln 逐项验证）
    ↓
【自动】API 文档生成（精简模式）
    ↓
【自动】集成测试脚本生成 + 执行 → 通过/失败报告
    ↓
【自动】单元测试生成（跳过 IQueryable 依赖的方法，标注需集成测试）
    ↓
人只需在确认节点做决策
```

**这一级需要：2 个新 Agent（`env-checker` + `integrity-checker`，确定性检查为主，token 消耗低）+ Agent prompt 强化。**

### 第三级：需求→代码（可达 ~1h，省 90%）

```
人脑风暴需求要点 → AI 生成需求 spec → 人确认 → AI 生成详细设计 spec
    → 人确认 → AI 全自动编码+测试+文档
```

**瓶颈不在"能不能做到"，而在于：**
- 需求 spec 的质量取决于人能否清晰表达
- 详细设计的架构决策（ADR）需要人的判断
- 这个瓶颈**不是技术瓶颈，是沟通瓶颈**

### 天花板：为什么省不到 95% 以上

| 不可替代的部分 | 占比 | 原因 |
|------|:--:|------|
| **需求定义** | ~5% | "出入库批次号要区分入库和出库"——这必须人说 |
| **架构决策** | ~3% | ADR-001 并发不超扣用原子 UPDATE 还是乐观锁？需要人判断 |
| **验收确认** | ~2% | 原型好不好用、文档够不够清晰——最终是人说了算 |

**Type B 的 AI 极限：省 ~90%，人仍然需要投入 ~1-2h 做需求和架构决策。**

## 四、两种类型对比总结

```
Type A (跨模块修复)
  ├── 当前：70h
  ├── 不改 harness，改行为：40h（省 43%）
  ├── 加轻量工具（Skill/Script）：25h（省 64%）
  ├── 改造 Agent prompt：18h（省 74%）
  └── 天花板：~18h，AI 做不了的是"根因判断 + 架构决策 + 业务规则确认"

Type B (新模块生成)
  ├── 当前：~12h
  ├── Agent 内置检查（不改架构）：~6h（省 50%）
  ├── 全自动流水线（加 Agent+check）：~2h（省 83%）
  └── 天花板：~1h，AI 做不了的是"需求定义 + 架构决策 + 验收确认"
```

## 五、关键洞察

### 1. Type A 和 Type B 需要不同的优化策略

Type B 的优化方向是**消除遗漏**（确定性规则 → Agent prompt 中的 checklist），Type A 的优化方向是**扩大视野**（不确定性问题 → 搜索影响范围 + 根因分析）。

给 Type B 加一个 `integrity-checker` agent 很值（每次省 2-3h 的遗漏修复时间）。给 Type A 加同样的 agent 效果有限——Type A 的问题不是"漏了什么"，而是"没意识到两个模块是耦合的"。

### 2. 不是所有问题都需要 Agent

| 问题 | 最合适的工具 | 为什么 |
|------|------------|------|
| 检测 .csproj 引用遗漏 | **Script**（bash grep） | 规则确定，不需要推理 |
| 跨模块影响范围搜索 | **Skill**（给 Agent 用的指令） | 需要理解"什么样的代码算相似"，但不需要独立 Agent |
| 环境健康检查 | **Script**（`init.sh --quick-check`） | 结果确定（pass/fail），不需要推理 |
| 根因分析 | **Agent**（需要对比多个文件、理解业务逻辑） | 需要推理和判断 |
| 回归测试生成 | **Skill**（嵌入 coder agent 的工作流） | 模板化程度高，可以写成指令 |

### 3. 天花板不是技术决定的

Type A 从 70h → 18h，Type B 从 12h → 1h。再往上省不动，不是因为 AI 不够聪明，而是因为**剩下的时间花在"人想清楚要什么"上了**——这不是 AI 能替代的。
