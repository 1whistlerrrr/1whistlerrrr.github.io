### 跨模块的"弱引用"（存 ID，无 FK）

有 4 个字段指向其他模块的表，但只存 ID + 冗余名称，不建外键：

| 字段 | 指向 | 所属模块 | 说明 |
|------|------|---------|------|
| `warehouses.responsible_user_id` | `users.id` | UserService | 只存 ID，冗余 `responsible_user_name` |
| `warehouses.responsible_org_id` | `org_dict.id` | 组织服务 | 只存 ID，冗余 `responsible_org_name` |
| `inout_orders.responsible_user_id` | `users.id` | UserService | 同上，冗余 `responsible_user_name` |
| `warehouses.video_device_ids` | 视频设备 ID | VideoMonitoring | 只存 `bigint[]`，前端 `VideoMonitor` 组件直接消费 |

**这不建 FK 是刻意的**（spec §3.2 T3 注释明确写了"跨模块弱引用，无 FK"），原因：
- 跨模块的强耦合会导致部署/测试复杂度（加载 UserService 才能建 MaterialSafety 表）
- 人员/组织数据可能来自外部同步，ID 不一定存在于当前库
- 冗余名称字段（`responsible_user_name`）保证即使源数据变化，历史记录仍可读

**结论**：这个模块在数据库层面完全自包含，不依赖其他模块的表结构。

这个错误是 Npgsql 类型解析问题。`DataTypeName '-'` 表示 Npgsql 无法确定某个列的 PostgreSQL 数据类型，但代码却尝试把它当 `String` 读取。

在你的代码中，最可能触发的地方是 **`InventoryListAsync`**（Service 第 652-673 行）：

```csharp
// 这个 LINQ join 产生匿名类型 { ledger, material }
var query = from ledger in _ledgerRepo.Query().Where(...)
            join material in _materialRepo.Query() on ... equals ...
            select new { ledger, material };

// PagingFetch 对匿名类型执行 CountAsync/ToListAsync
var paged = await query.PagingFetch(cleanOption, ...);
```

**原因**：LINQ `join` 两个 `IQueryable` 生成的 SQL 中，多个表的列混在一起。当 `PagingFetch` 调用 `CountAsync()` 或 `ToListAsync()` 时，匿名类型的投影可能让 Npgsql 拿到一条 metadata 为空的列，尝试映射到匿名类型的 String 属性时就报这个错。

**但实际上测试中这个接口是正常返回的**（返回了 3 条库存数据）。这个错误可能是：
- 某些特定数据条件下触发（如 JOIN 结果为空、分页边界条件）
- Npgsql 版本 bug
- 或者是你在调试时用 `GetString()` / `ExecuteReader` 直接读了某列？

### 第 44 轮

**👤 提问**

allowNoPaging: false这是什么意思 举例解释

**🤖 回答**

`allowNoPaging` 是 `PagingFetch` 扩展方法的一个**安全开关**，控制是否允许客户端要求"不分页，全部返回"。

### 源码逻辑（`FrontQL/Extensions/PagedQuery.cs:124-126`）

```csharp
if (option.NoPaging && allowNoPaging == false) {
    throw new NotSupportedException("NoPaging is not allowed.");
}
```

### 举例

客户端请求体里有个 `noPaging` 字段：

```json
// 正常分页（默认）
{ "page": 1, "pageSize": 20, "noPaging": false }
// → 返回第1页，每页20条

// 不分页，全量返回
{ "noPaging": true }
// → 跳过 Skip/Take，把符合条件的数据一口气全返回
```

`allowNoPaging` 控制的是：**服务端允不允许客户端用 `noPaging: true`**。

| allowNoPaging | 客户端发 `noPaging: true` | 结果 |
|:--:|------|------|
| `false` | 要全量 | **抛异常** `NotSupportedException`，请求被拒绝 |
| `true` | 要全量 | 正常返回全部数据，不走分页 |

### 为什么要设 false

防止客户端无意中请求百万级数据把服务打崩。字典表通常只有几十条，所以可以放开；流水表可能几十万条，应该禁止。

### 对比

```
MaterialDict/List:  allowNoPaging: true   ← 3条数据，全量返回没问题
InoutOrder/List:    allowNoPaging: false  ← 可能有上万条流水，禁止全量
```
