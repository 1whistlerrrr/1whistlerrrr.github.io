# SZXA-HARNESS 工作流分析与改进报告

> 分析来源：物资安全管控模块完整开发过程（两段对话共 93 轮交互）
> 会话1：模块设计/命名/PageModule/项目总览（6 轮，07-20）
> 会话2：后端完整编码→测试→验证→迭代（87 轮，07-14 至 07-21）
> 分析时间：2026-07-22

| 6 | **单元测试编写** | 测试 | 4~6 | Agent 创建 2205 行测试文件 → 76 用例，56 通过，20 跳过（PagingFetch 需 EF Core provider） | ⚠️ 中 | 覆盖所有 CRUD 参数校验、唯一性检测、边界条件；跳过的 20 个因纯 Moq 无法模拟 EF Core 异步操作 | AI 应自动判断哪些用例适合单元测试 vs 集成测试，避免技术限制导致的 skip | 2~3 |

## 二、人机交互分析

### 2.1 人的缺陷与改进（提示词方面）

| # | 问题 | 表现 | 改进建议 |
|---|------|------|---------|
| 1 | **提示词过于简略** | 多次只发一句话甚至几个词（"出入库管理"、"allowNoPaging: false这是什么意思"），缺少文件路径、上下文、期望效果 | **改进**：使用"我要在 [文件路径] 中做 [具体操作]，因为 [原因]，期望结果是 [具体效果]" 的结构化提示词模板。对于概念性问题，附上"我的理解是 X，对吗？" |
| 2 | **需求前置不足** | 编码完成后才提出批次号区分出入库、noPaging 支持等需求，导致返工修改 | **改进**：在详细设计阶段用 checklist 逐项确认——"还有没有类似的边界场景？""所有字段是否都需要支持全量返回？""编号规则是否需要区分类型？" |
| 3 | **对基础设施理解不足** | 不知道解决方案(.sln)是什么、不知道怎么管理分支导致团队协作受阻、不知道 DI 容器生命周期 | **改进**：利用 AI 做知识补充——在遇到不懂的概念时追问到底（如第 56 轮追问".sln 是什么"），同时建议在项目总览中加入"开发环境基础概念"章节 |
| 4 | **测试配置准备不充分** | 要求测试时没有同时提供 token/ProjectId/服务地址，需要 AI 追问 | **改进**：在启动测试前准备好"测试环境信息卡片"（地址 + token + ProjectId + 时区），作为测试请求的标配附件 |
| 5 | **验证依赖 AI 自检** | 多次要求 AI "检查是否与设计一致"，但缺少自己的验收标准 | **改进**：建立个人验收 checklist（编译通过 + 关键 API curl 验证 + 至少看 3 个关键文件的 diff），不纯粹依赖 AI 自述 |
| 6 | **任务切换缺乏追踪** | 在第 47 轮才发现代码没合到 develop，在第 48 轮切分支时遇到 stash 冲突 | **改进**：利用 `.claude/current-task` 和 `计划进度文档.md` 跟踪当前在哪个分支、哪个阶段，每次切换时先检查状态 |

### 2.2 AI 的缺陷与常见错误

| # | 问题类型 | 具体表现 | 根因 | 改进方法 |
|---|---------|------|------|---------|
| 1 | **遗漏关键配置** | Phase 2 在 `ModulesLoader.cs` 注册了模块，但漏掉 `HostAPP.csproj` 的 `<ProjectReference>`，导致编译失败（CS0246） | Agent 没有"新增模块 → 必须添加项目引用"的 checklist | **Agent prompt 内置检查清单**：新增模块时强制检查：① `.csproj` 引用 ② `ModulesLoader` 注册 ③ `AppDbContext` DbSet ④ DDL ⑤ `.sln` 包含 |
| 2 | **枚举序列化假设错误** | API 文档将 `inoutType` 标注为 `number`（枚举数字），实际项目全局配置了 `JsonStringEnumConverter`，返回的是 camelCase 字符串（`"in"`/`"out"`） | Agent 基于通用 C# 惯例假设，未检查项目实际配置（`AccessCore/Shared/Json.cs`） | **Agent 必须先读全局配置再写文档**：生成 API 文档前强制读取 `Json.cs` 中的序列化配置、`ApiResult.cs` 的响应格式、`EnumModelBinder` 的 query 参数处理 |
| 4 | **过度设计（内存层不必要）** | 批次号生成第一版用了 `ConcurrentDictionary` + 内存缓存，用户质疑后才改为纯 DB 查询（`SELECT MAX` + 直接 +1） | Agent 倾向于"多加一层保障"，但没评估该场景是否真的需要 | **设计决策遵循 YAGNI 原则**：在 Agent prompt 中加入"优先选择最简方案，除非 spec 明确要求或数据量/并发量证明需要缓存" |
| 5 | **测试策略失配** | 76 个单元测试中 20 个跳过，因为 `PagingFetch`（`CountAsync`/`ToArrayAsync`）需要真实 EF Core provider，纯 Moq 无法模拟 | Agent 用单元测试框架写了集成测试场景的用例 | **Agent 在写测试前应先分类**：分析每个被测方法，判断它属于纯逻辑（→ 单元测试，Moq 可覆盖）还是依赖 EF Core IQueryable（→ 集成测试，需 InMemory/Testcontainers），输出分类表后再写代码 |
| 6 | **文档过于冗长** | API 接口文档被用户评价"废话太多了"，测试环境配置章节包含了详细的获取方式、检查清单等，用户只需 3 个 Header + 一个 curl 示例 | Agent 遵循"越完整越好"的文档模板 | **文档生成支持精简/完整双模式**：规范中定义"AI 自用完整版"（含每步详解）和"人类速查精简版"（仅必填项 + 示例），默认生成后者 |
| 7 | **对项目全局状态感知不足** | 无法预先判断 DI 生命周期冲突（Singleton→Scoped）、DB 连接池耗尽等问题，只能在启动失败后被动分析 | Agent 在 worktree 隔离环境中编码，无法感知 HostAPP 的整体状态 | **引入"环境健康检查"门控**：在编码前/后自动执行编译检查 + DI 冲突检查 + 数据库连通性检查，作为过程 4 的强制步骤 |
| 8 | **重复执行相同检查** | 设计一致性验证跑了至少 4 轮（第 27/28/29/30/31/32 轮），多次返回"0 差异"但用户仍要求再查 | Agent 没有返回"上次已检查通过"的缓存意识 | **检查结果应可追溯**：每次检查输出带上时间戳和 hash，后续重复请求时先展示上次结果，询问"上次检查 N 分钟前已通过，是否仍需重查？" |
| 9 | **低估跨模块影响的排查难度** | 将 HostAPP 启动失败简单归因于"其他模块既有问题"，但用户花了多轮（第 49~54 轮）才确认根因 | Agent 排查能力受限于 worktree 隔离，无法修改其他模块代码 | **问题分级输出**：排查结果按"本次改动导致 / 既有问题但本次暴露 / 环境问题 / 无法判断"四级分类，每类给出具体修复建议（而非仅说"与我无关"） |
| 10 | **代码注释不规范** | Service 代码中出现了"// 这是一条注释"风格的中文注释，与项目既有注释风格不一致 | Agent 没有读取项目注释规范 | **coder agent 在编码前必须读取至少 2 个已有 Service 文件**（已有规则但未强制执行），提取注释风格、using 排序、region 命名等隐性约定 |

## 三、Harness 工作流改进方案

### 3.1 过程 4（编码阶段）应增加的前置门控

当前流程：

```
task-planner → backend-coder → frontend-coder → code-reviewer → evaluator
```

建议改进为：

```
task-planner
  ↓
【新增】环境健康检查 agent（env-checker）
  ├── 编译主项目 HostAPP（确认基线状态）
  ├── 检查 DI 容器所有注册（无 Singleton→Scoped 冲突）
  ├── 检查 DB 连通性（连接池可用）
  └── 检查目标分支状态（git status / 与 develop 的 diff）
  ↓
backend-coder（worktree 隔离）
  ├── 【新增】编码前：自动读取 ≥2 个同类文件作为参考（提取注释风格、using 顺序、region 命名）
  ├── 【新增】编码后自检：dotnet build + 检查新增项目引用是否完整
  └── 【新增】输出"新增/修改文件清单"（标注是否新增了 .csproj 引用、是否注册了 Module）
  ↓
【新增】编码完整性校验 agent（integrity-checker）
  ├── 逐项检查：.csproj 引用 / ModulesLoader 注册 / AppDbContext DbSet / DDL / .sln 包含
  ├── 对照详细设计 spec 检查：API 数量 / URL / Method / 参数位置
  └── 产出"完整性检查报告"（分 Critical/Warning/Info）
  ↓
code-reviewer → evaluator
```

### 3.2 Agent Prompt 需要强化的内置检查清单

**coder agent 强制检查项**（在 agent 定义文件的 prompt 中直接写入）：

```
编码前检查：
  ☐ 已读取 ≥2 个同类文件作为风格参考
  ☐ 已读取项目全局配置（Json.cs 序列化、ApiResult 格式、EnumModelBinder 规则）
  ☐ 已确认目标文件夹位置和命名约定

编码后检查（自检）：
  ☐ dotnet build 通过（0 Error）
  ☐ 新增项目 → 已在 HostAPP.csproj 添加 ProjectReference
  ☐ 新增项目 → 已在 ModulesLoader.cs 注册 Module
  ☐ 新增实体 → 已在 AppDbContext.cs 添加 DbSet + OnModelCreating
  ☐ 新增表 → 已在 upgrade.sql 追加 DDL（含索引、约束、COMMENT）
  ☐ 新增项目 → 已加入 .sln 解决方案
  ☐ 新增 API → Controller URL/Method/参数与 spec 一致
  ☐ ViewModel 字段名/类型/[Required] 与 spec 一致
```

### 3.3 新增 agent 类型建议

| Agent | 触发时机 | 职责 |
|-------|---------|------|
| `env-checker` | 过程 4 启动前 | 环境健康检查（编译基线、DI 状态、DB 连通性、分支状态） |
| `integrity-checker` | coder agent 完成后 | 编码完整性校验（.csproj/Module/DDL/sln 全链路检查） |
| `test-classifier` | 单元测试编写前 | 分析每个方法，判断适合单元测试还是集成测试，输出分类表 |

### 3.4 Skill 体系改进

| 改进点 | 当前状态 | 建议 |
|--------|---------|------|
| API 文档生成 | 默认"完整模式"，产出冗长 | 新增 `--mode=brief` 参数，人类速查版只需：必填 Header 表格 + curl 示例 + API 编号总览 |
| 测试生成 | 单元测试和集成测试混在一起 | 新增 `test-classifier` skill，在写测试前先分类，避免 26% skip 率 |
| 设计一致性验证 | 需人工多次触发 | 作为 `integrity-checker` agent 的输出物，编码完成后自动执行 |
| 需求完整性检查 | 无 | 新增 `requirement-completeness` skill，在过程 1 结束时逐项检查：编号规则 / 筛选方式 / 分页策略 / 是否区分类型 |

### 3.5 Hook 改进

| Hook | 当前 | 建议增加 |
|------|------|---------|
| `pre-edit-guard.sh` | 仅提醒"不要直接写产品代码" | 增加检测：如果编排者正在操作 `.csproj` 或 `ModulesLoader.cs` 且不属于 harness 基础设施操作 → 拦截并提醒"这应该由 coder agent 完成" |
| `trace-writer.sh` | 记录所有工具调用 | 增加**重复调用检测**：如果同一检查（如设计一致性验证）在 10 分钟内被触发 2 次以上 → 注入提醒"此检查最近已执行，结果在 trace 中" |

### 3.6 核心工作流原则总结

1. **编码前必读参考代码**（已有规则，需在 agent prompt 中做硬性检查）
2. **编码后必跑完整性检查**（新增，解决 .csproj 引用遗漏等低级遗漏）
3. **环境基线性检查前置**（新增，不依赖启动失败后被动排查）
4. **测试先分类后编写**（新增，消除因技术限制导致的无效测试）
5. **检查结果可缓存、可追溯**（新增，避免重复检查浪费 token 和时间）
6. **文档双模式输出**（改进，区分"AI 自用完整版"和"人类速查精简版"）
7. **需求完整性 checklist 前置到过程 1**（改进，减少编码后返工）
