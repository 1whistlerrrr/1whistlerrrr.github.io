## 二、这 70 小时花在哪了——问题分类

### 2.1 按问题类型拆分

| # | 问题类型 | 代表会话 | 轮次占比 | 根因 |
|---|---------|---------|:--:|------|
| 1 | **时区处理不一致** | WorkerOnline 69轮 + DateTimeOffset 15轮 | ~25% | 两个模块对时区的处理方式不同（WorkerOnline 用 UTC，Attendance 用当地时间+timezone），AI 一开始没发现这种不一致 |
| 2 | **工时计算算法反复** | 跨天回溯、haton 隔断逻辑、needLookback 去除 | ~20% | 算法需求是边修边明确的，不是一开始就设计好的——"去掉回溯""按零点隔断""haton 也按天隔断"是逐步发现的 |
| 3 | **Npgsql 类型映射错误** | `Reading as 'String' not supported for DataTypeName 'bigint'` | ~10% | EF Core + Npgsql 的 LINQ 投影类型推断 bug，AI 多次犯同样错误 |
| 4 | **DI 生命周期混用** | Singleton→Scoped DbContext 冲突，Attendance 模块改造 | ~15% | 既有代码用 Singleton 注册依赖 Scoped DbContext 的 Service，AI 在最初编码时就埋下了这个坑 |
| 5 | **性能优化试错** | 冷库查询、批量操作、连接池耗尽 | ~10% | AI 建议的优化方案被用户撤回，性能问题反复出现 |
| 6 | **两个接口结果不一致的排查** | WorkerOnline vs Segments 对比 | ~10% | 两个模块在不同时间由不同 agent 编写，逻辑不统一 |
| 7 | **测试与验证** | Python 脚本、curl 测试、单元测试 | ~10% | 大量手动测试，反复"测试→发现问题→修改→再测试" |

## 三、人的问题：有什么不足，如何改进

### 3.1 需求是"边修边发现"的，不是一开始就明确的

**表现**：
- 第 1 轮："为什么 WorkOnline 返回段之前的值"——只发现了一个 symptom
- 第 12 轮："把 OnlinePeriods 的算法合并到 Segments"——才意识到应该统一算法
- 第 47 轮："按照 timezone 进行偏移，按照零点进行隔断"——才发现跨天边界问题
- 第 50 轮："去掉 needLookback"——才发现回溯逻辑是多余的

**问题本质**：你把"发现 bug → 定位根因 → 设计修复方案 → 实施"这四步全部交给了 AI，而不是你先做好前三步再让 AI 实施。

**改进方法**：
- 发现 bug 后先不急着让 AI 改代码，先和 AI 做一轮"根因分析对话"——只问"为什么会这样，可能的原因有哪些"
- 把分析结果写成 2-3 句话的**修复方案**，再让 AI 执行
- 范例提示词模板：
  ```
  我发现了问题：[具体 symptom]。
  我推测根因是：[你的猜测]。
  请帮我验证这个推测是否正确，然后给出修复方案（先不改代码）。
  ```

### 3.2 概念不懂时"边修边学"，拖慢了修复节奏

**表现**：
- 在修时区 bug 的过程中问"大小写有关系吗"（第 9 轮）
- 在修 DI 冲突时学了 5 轮 Singleton vs Scoped（第 17~19、25~29 轮）
- 在讨论解决方案时问".sln 是什么"（另一个会话第 56 轮）

**问题本质**：你把"修 bug"和"学习概念"混在同一会话里了。对于紧急 bug 修复，这就像修车的时候翻说明书。

**改进方法**：
- **分离学习和修复**：遇到不懂的概念，记下来，开一个独立会话专门问——不占用修复会话的认知带宽
- 学习型会话用 skill `grill-me` 或直接说"教我 X，用举例的方式，不要改任何代码"

### 3.3 测试手段原始，验证周期长

**表现**：
- 大量手动 curl 命令
- Python 测试脚本（`api_test.py`, `regenerate_records.py`）写完后又花很多轮调试脚本本身
- 反复"改代码 → 编译 → 启动服务 → curl 测试 → 发现问题 → 再改"

**问题本质**：没有"一键回归测试"的能力。每次改完代码都要手动验证所有相关接口。

**改进方法**：
- 在第一次发现问题时就让 AI 生成一个**自动化回归测试脚本**（覆盖所有相关接口的各种场景）
- 每次修改后跑一次脚本，30 秒内知道有没有 break 其他东西
- 对 AI 说：_"帮我写一个测试脚本，覆盖 [接口1]+[接口2] 的所有场景（正常值、边界值、时区切换），每次修改代码后我跑一次这个脚本就能验证没有回归"_

### 3.4 上下文频繁切换

**表现**：
- DateTimeOffset 会话中（15 轮），第 4~7 轮突然跳到 SSO 第三方登录问题，第 8 轮才回到 attendance
- WorkerOnline 会话中穿插了 generate、record list 等不直接相关的话题

**问题本质**：一个会话里塞了太多不相关的任务，AI 和人都容易丢失主线。

**改进方法**：
- **一个会话 = 一个任务**，遇到新问题开新会话
- 如果当前会话已经很长（>20 轮），考虑 handoff：让 AI 总结当前进度，开新会话继续

### 3.5 对 AI 修改缺乏信心——"会不会把原有逻辑改了"

**表现**：
- 第 9 轮："我其实担心的是，这个修改会不会把原有的逻辑给改动了"
- 多次要求 AI "对比两个接口的代码逻辑"、"查看代码进行修改"、"检查是否真的是最终结果"

**问题本质**：你不信任 AI 的修改不会引入回归。这本身是理性的——但反复验证的成本很高。

**改进方法**：
- 要求 AI 在每次修改前先输出**影响范围分析**（"这个改动会影响 X 个文件的 Y 个方法，具体是..."）
- 修改完成后要求 AI run 所有相关测试
- 如果测试覆盖率不够，**先补测试，再修 bug**

## 四、AI 的问题：经常犯什么错，如何改进

### 4.1 类型映射错误（Npgsql `DataTypeName` 异常）

**表现**：
- `Reading as 'System.String' is not supported for fields having DataTypeName 'bigint'`
- `Reading as 'System.String' is not supported for fields having DataTypeName '-'`
- 同一类错误在 WorkerOnline、AttendanceRecord 两个模块都出现过

**根因**：AI 在写 LINQ 投影（`Select new { ... }`）时，对 Npgsql 的列类型推断机制不够了解。匿名类型的属性类型推断依赖 EF Core 的元数据，当 JOIN 或复杂投影时可能拿到错误的 DataTypeName。

**改进方法**：
- **Agent prompt 中增加 Npgsql 类型映射规则**：禁止在 LINQ 投影中直接用 `string` 属性接收可能是 `bigint` 的数据库列
- **事后检查**：每次修改涉及 LINQ 查询后，自动跑一次相关的 API 测试

### 4.2 DI 生命周期混用（Singleton→Scoped）

**表现**：
- `AttendanceOnlinePeriodQueryService` 被注册为 Singleton，但依赖 Scoped 的 `AppDbContext`
- `IAttendanceHeartbeatQueryService` 也是同样问题
- 修改后引入了 `IServiceProvider` + `CreateScope` 的 anti-pattern，然后又改回直接注入 `AppDbContext`

**根因**：AI 在最初编写这些 Service 时，没有遵循项目的 DI 惯例。既有代码已经有 Singleton 注册，AI 沿着既有模式继续写，没有发现这个设计问题。

**改进方法**：
- **Agent prompt 增加 DI 规则**：依赖 `AppDbContext` 的 Service **必须**注册为 Scoped（或 Transient），绝不 Singleton
- **编码后检查**：`integrity-checker` agent 应该 grep 所有 `AddSingleton<ISomething, Something>` 并检查 Something 的构造函数是否注入了 Scoped 服务

### 4.3 算法修改不完整——改了一个模块漏了另一个

**表现**：
- 修改了 WorkerOnline 的 hatoff 处理逻辑，忘了同步修改 Attendance 模块中对应的逻辑
- 去掉了 WorkerOnline 的 needLookback，但 WorkerOnlinePeriodQueryService 还在用回溯逻辑
- 修改了时区处理方式，但只在 Segments 改了，Generate 方法还在用旧逻辑

**根因**：两个模块（WorkerOnline 和 AttendanceRecord）有相似的算法，但 AI 没有意识到它们在逻辑上的耦合关系。

**改进方法**：
- **修改前强制做影响范围分析**：让 AI 先搜索所有调用相同方法/使用相似模式的位置，输出"需要同步修改的清单"
- **增加跨模块的搜索 skill**：_"当我要求修改算法时，先搜索整个代码库中有没有使用相同模式的其他位置，列出后让我确认是否全部都要改"_

### 4.4 性能优化不靠谱

**表现**：
- 建议的"批量优化方案"被用户撤回
- 冷库查询的讨论绕了很多圈（"现在不会查冷库了吧"→"为什么不太可能是瓶颈"→实际的瓶颈不在那里）
- 对"为什么慢"的判断不准

**根因**：AI 没有 profiling 数据，只能猜测性能瓶颈。猜错是必然的。

**改进方法**：
- **先测量再优化**：让 AI 先在代码中加计时日志，跑一轮真实数据，拿到各步骤的耗时分布，再针对性优化
- 对 AI 说：_"不要猜瓶颈在哪，先帮我加计时日志定位最慢的步骤"_

### 4.5 解释型回答太多，直接行动不够

**表现**：
- 很多轮次的 AI 回答是"分析原因 + 列方案"，等用户确认后才行动
- 像 Singleton vs Scoped 这种学习型对话，AI 应该检测到这是概念问题，建议用户读文档或开独立学习会话

**改进方法**：
- 对于明确的操作（如"修改类型"、"统一时区处理"），AI 应该直接执行 + 报告，而不是先列 3 个方案让用户选
- 对于知识型问题，AI 应该简短回答 + 建议"如果需要深入理解，可以开一个学习会话"

## 五、核心改进建议（Top 5）

### 1. "根因分析"前置——修 bug 前先理解

不要上来就改代码。第一轮交互应该是：
> "我发现 [symptom]。帮我分析可能的原因，列出所有可能性，然后逐一排除。先不改代码。"

这样 69 轮的 WorkerOnline 会话可能压缩到 20 轮以内。

### 2. "一键回归测试"——每次修改后自动验证

bug 修复的第一个产出不是代码修改，而是**回归测试用例**。

> "帮我先把当前所有相关接口的正确行为写成测试脚本。之后每次修改代码，我跑一次这个脚本就知道有没有引入回归。"

### 3. "影响范围分析"——修改前先搞清楚会动哪些

> "在改代码之前，先搜索所有使用 [方法名/模式] 的位置，告诉我这个改动会影响哪些文件、哪些方法。"

### 4. "学习与修复分离"——不懂的概念开独立会话

修 bug 时遇到不懂的概念 → 记下来 → 继续修 bug → 修完之后开一个学习会话专门补概念。

### 5. "测试手段自动化"——别再手动 curl 了

第一次测试就生成完整的测试脚本（Python/curl 合集），覆盖：
- Happy path（正常值）
- 边界条件（跨天、跨月、时区边界）
- 异常场景（不存在的 personId、空数据）
- 回归对比（WorkerOnline vs Segments 结果一致性）
