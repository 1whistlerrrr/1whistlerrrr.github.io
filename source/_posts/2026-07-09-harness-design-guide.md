## 核心理念

Harness 不是"用 AI 写代码"，而是一套**完整的 AI 编排系统工程**——用 **过程路由 + 专职 Agent + 对抗审查 + Hook 拦截** 把 feature 从需求到上线的全生命周期自动化。

---

## 一、过程路由：把"人驱动流程"变成"AI 驱动流程"

传统开发中，需求→设计→计划→编码→审查→验证靠人为推动和交接。Harness 将其固化为 **严格顺序的过程**，由编排者自动路由：

```
过程1: 需求规格化 → 原型确认
过程2: 全栈详细设计
过程3: 实施计划
过程4: 编码 → 审查 → 验证
```

**设计要点**：
- 每个过程的输入和产出必须**结构化、可校验**——前一过程的产出是后一过程的硬依赖
- 过程之间由编排者（Orchestrator）自动判断门控条件，而非等待人工确认
- 每个过程可以独立迭代，但不可跨过程跳转

---

## 二、专职 Agent 分工：拆分角色，而非单打独斗

不要设计一个"全能 Agent"。将开发工作拆分给 **多个高度专职化的 Agent**，各司其职：

### Agent 角色矩阵

| 阶段 | Writer Agent | Reviewer Agent |
|------|-------------|----------------|
| 需求 | `spec-writer` — 将需求要点转化为结构化文档 | `spec-reviewer` — 审查需求的完整性和无歧义性 |
| 设计 | `designer` — 全栈详细设计（API/数据库/流程） | `design-reviewer` — 确保设计符合架构规范 |
| 计划 | `plan-writer` — 将设计转化为精确到文件路径的实施计划 | `plan-reviewer` — 确保计划可执行 |
| 编码 | `coder` — 按计划编码 + 写测试 | `code-reviewer` — 多维度代码审查 |
| 验证 | `evaluator` — E2E 端到端验证 | — |
| 运维 | `issue-analyzer` — 线上问题归因分析 | — |

外加一个 **`task-planner`** 将实施计划拆解为子任务 DAG。

**设计要点**：
- 每个 Agent 的职责边界要**窄而深**——一个 Agent 只做好一件事
- Writer + Reviewer 成对出现，形成对抗审查
- 编码 Agent 按技术栈拆分（前端/后端/移动端），各自拥有对应技术栈的专属 skill
- 每个 Agent 的 system prompt 中内嵌该角色的**完整检查清单**

---

## 三、GAN 对抗审查循环：AI 互审，质量门控自动化

这是 Harness 最核心的质量保障机制。每一对 writer + reviewer 形成**对抗生成式循环**（建议上限 3 轮）：

```
writer 产出 → reviewer 审查 → 发现问题则回写 →
writer 修复 → reviewer 再审查 → 直到通过或判定无法在此阶段解决
```

**设计要点**：
- 审查结果必须**分级**（Critical / High / Medium / Low），便于判断是否阻断流程
- 设定硬性退出条件（如"0 Critical + 0 High"才算通过）
- 设定最大迭代轮次（建议 3 轮），超过则标记为需人工介入
- reviewer 的 prompt 要包含"逐项检查清单"，确保审查不遗漏

---

## 四、Coder Agent 的标准工作流

Coder Agent 不应只是"接收 prompt → 输出代码"，而应遵循固定的**工程化工作流**：

```
1. 自动读取所有规范文档（架构约束、代码规范、测试策略）
2. 自动读取详细设计 spec（API 契约、数据模型）
3. 自动读取至少 2 个已有实现作为参考范例
4. 编写 DDL/数据迁移 → 编写业务代码 → 编写测试 → 运行验证 → 生成 API 文档
5. 强制退出条件：构建 0 错误 + 测试 0 失败
```

**设计要点**：
- Agent 必须**先读后写**——读规范、读设计、读参考实现，才能写代码
- 设置**硬性退出门槛**（编译通过 + 测试通过），不满足则 Agent 自行修复迭代
- 产出物包括代码、测试、文档——三位一体，不可缺漏

## 五、Code Reviewer 的多维度审查框架

Code Reviewer 不应只给笼统评价，而应按**固定维度逐项检查**：

### 维度 1：代码质量（通用检查项）
- 方法/函数长度是否合理
- 是否存在重复代码
- 依赖注入是否正确
- 是否存在 N+1 查询
- 异常处理是否完善
- 时区/编码等隐式假设是否安全
- null 保护是否到位

### 维度 2：设计符合情况（与设计 spec 对比）
- 数据库迁移脚本是否已执行
- 模块注册/路由配置是否完整
- API 路径和参数是否与设计一致
- 数据模型/响应字段是否匹配契约
- 集成事件/消息契约是否正确
- 外键策略、事务边界是否合理

### 维度 3：任务完成度
- 子任务是否全部覆盖
- 测试覆盖率是否达标
- 是否有遗漏文件或未实现接口
- 与上下游接口是否对齐

**设计要点**：
- 每个维度下要有**具体到文件和行号**的检查项，而非模糊评价
- 审查结果结构化输出，可直接作为修复依据
- 审查维度可根据项目特点增减

## 六、Hook 拦截层：模型无法绕过的工程护栏

Hook 在 AI Agent **外部**运行，AI 无法绕过或忽略。这是将规范从"建议"变成"强制"的关键机制：

| Hook 时机 | 作用 | 示例 |
|-----------|------|------|
| `SessionStart` | 注入上下文 | 自动注入当前分支、仓库状态、环境变量 |
| `UserPromptSubmit` | 校验任务 | 检查当前任务和迭代轮次是否匹配，防止在错误上下文中工作 |
| `PreToolUse` | 拦截违规操作 | 拦截编排者直接写产品代码（必须通过专职 Agent） |
| `PostToolUse` | 记录审计 | 所有关键文件操作自动记录到 traces |
| `SubagentStart/Stop` | 记录对话 | Agent 的完整对话过程自动存档 |

**设计要点**：
- Hook 是**外部约束**，不依赖 AI "记住"规则
- 将架构约束同时编码进 Agent prompt（软约束）和 Hook 逻辑（硬约束）
- 所有关键操作自动留痕，支持事后审计和回溯

---

## 七、Skill 模块化：可复用的能力单元

将常用能力封装为**可复用的 Skill**，Agent 按需调用：

```
Skill 分类建议：
├── 编码类：单元测试模板、ETL 模板、重构工作流
├── 审查类：按维度的审查清单（安全/风格/错误处理/性能）
├── 部署类：部署前检查清单、安全提交模板
└── 运维类：日志分析、问题归因
```

**设计要点**：
- 每个 Skill 应独立、可测试、有明确的输入输出
- Skill 之间可组合但不应耦合
- Skill 内容应匹配团队的技术栈和规范

---

## 八、Manifest 驱动配置：一致性保障

所有 Agent、Skill、Hook、配置通过一个 manifest 文件统一管理：

```json
{
  "agents": { ... },    // Agent 定义（角色、prompt、可用工具）
  "skills": { ... },    // Skill 定义（名称、触发条件、模板）
  "hooks": { ... },     // Hook 定义（时机、拦截逻辑）
  "constraints": { ... } // 架构约束（代码规范、技术限制）
}
```

**设计要点**：
- 一键初始化脚本（`init --sync`）同步所有配置
- 团队任何人加入项目，跑一次脚本获得完全一致的开发环境
- Agent 行为的一致性由配置驱动，而非每个人的 prompt 调试

---

## 九、架构约束的双重编码

项目级的架构约束（如技术选型限制、代码风格、数据库规范）应**同时在两个层面编码**：

| 层面 | 机制 | 效果 |
|------|------|------|
| Agent Prompt | 软约束 | Agent 编写代码时自觉遵循 |
| Hook 逻辑 | 硬约束 | 违规操作被 Hook 直接拦截 |

**示例约束类型**：
- API 风格（RESTful / RPC / GraphQL）
- ID 生成策略（自增 / UUID / 雪花）
- ORM 查询策略（自动 NoTracking / 禁止裸 SQL）
- 服务生命周期（Singleton / Scoped / Transient）
- 数据库范式要求
- 文件组织规范

**设计要点**：
- 约束要**具体可执行**，而非"代码要写得好"这种模糊要求
- 每个约束都要有明确的检查方式——Agent 如何自查、Reviewer 如何检查、Hook 如何拦截

## 十、全流程总结

```
人脑风暴需求要点
       ↓
  [过程1] spec-writer ←→ spec-reviewer (对抗 × N 轮)
       ↓  产出: 需求文档 / 验收标准
  [过程2] designer ←→ design-reviewer (对抗 × N 轮)
       ↓  产出: 详细设计 / API 契约 / 数据模型
  [过程3] plan-writer ←→ plan-reviewer (对抗 × N 轮)
       ↓  产出: 精确到文件路径的实施计划
  [过程4] task-planner → coder → code-reviewer → evaluator
              ↓              ↓            ↓           ↓
           子任务 DAG     编码+测试    多维度审查   E2E 验证
```

**Harness 设计的五个核心原则**：

1. **过程固化**：阶段不可跳过，门控自动化
2. **角色拆分**：专职 Agent 各司其职，对抗审查保证质量
3. **双重约束**：Prompt（软）+ Hook（硬），规范不被绕过
4. **结构化产出**：每阶段产出可校验、可追溯、可审计
5. **配置驱动**：Manifest 统一管理，一键同步，行为一致

## 附录：Harness 设计检查清单

开始设计你自己的 Harness 时，对照以下清单：

- [ ] 定义了哪些固定过程？每个过程的输入/产出/退出条件是什么？
- [ ] 每个过程有哪些专职 Agent？Writer 和 Reviewer 是否成对？
- [ ] 审查维度是否结构化？是否有分级（Critical/High/Medium/Low）和通过标准？
- [ ] 对抗循环的最大迭代轮次是多少？超限后如何升级？
- [ ] Coder Agent 的强制退出条件是什么（构建/测试/文档）？
- [ ] 有哪些架构约束？是否同时在 Prompt 和 Hook 中编码？
- [ ] Hook 在哪些时机拦截？拦截的逻辑是什么？
- [ ] 所有操作是否有审计日志（traces）？
- [ ] Skill 是否模块化、可独立测试？
- [ ] 是否有 Manifest 文件统一管理配置？
- [ ] 新成员加入项目的最小初始化步骤是什么（能否一键同步）？
