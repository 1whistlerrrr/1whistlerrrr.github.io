## 摘要

在优化考勤系统的 `RecalculateAll` 接口时，发现大量 QueryService 注册为 `Singleton`，但每个方法内部都通过 `CreateAsyncScope()` 手动借连接。本文记录了我们围绕"应该改成 `Scoped` 吗"展开的讨论，从 Singleton/Scoped 的本质区别、线程安全的误区，到选型决策框架，以及我自己还没完全理解清楚的问题。

---

## 一、背景：一个性能问题引出的架构讨论

考勤系统有一个运维接口 `RecalculateAll`，会遍历所有项目、计算所有人员的工时并写入数据库。

代码流程：

```
RecalculateAll
  → 每个项目并行 (Parallel.ForEachAsync)
    → 人员分块并行 (Task.WhenAll)
      → 每人调用 ComputeOnlineSegmentsAsync
        → GetBindingRangesAsync()           ← CreateAsyncScope() ①
        → GetGridPolygonsAsync()            ← CreateAsyncScope() ②
        → GetLocationAndWearingDataAsync()  ← CreateAsyncScope() ③
        → GetHeartbeatBitmapsAsync()        ← CreateAsyncScope() ④
```

**问题**：每算一个人就借还 4 次数据库连接。3188 人 × 4 = 12752 次连接借还。

这些 QueryService 注册为 `Singleton`，却又在每个方法里自己 `CreateAsyncScope()` 临时借 DbContext。为什么不直接注册为 `Scoped`、直接注入 DbContext？

## 二、Singleton 和 Scoped 到底是什么

### 2.1 生命周期对比

| | Singleton（单例） | Scoped（作用域） |
|---|---|---|
| 实例数量 | 整个应用只有 1 个 | 每个请求 / scope 各 1 个 |
| 存活时间 | 应用启动 → 关闭 | 请求进入 → 结束 |
| 类比 | 公司只有一个 CEO，所有人都找他 | 餐厅每桌配一个服务员，一对一服务 |
| 能直接注入 DbContext？ | ❌（DbContext 是 Scoped，会比 Singleton 先释放） | ✅（同生共灭） |

### 2.2 为什么需要两种生命周期？

**如果全部用 Singleton**：
- DbContext 从启动活到关闭。请求 A 查的数据还在内存里没保存，请求 B 又读到了 → 数据污染、内存膨胀、完全不可用

**如果全部用 Scoped**：
- 一个全局配置"最大重试次数 = 3"，十年不变，每个请求都新建实例从数据库重新读 → 白白浪费

所以设计上必须有区分：**活得长、无状态、全局共享** 用 Singleton；**活得短、带状态、请求隔离** 用 Scoped。

### 2.3 关键澄清：Singleton 不等于单线程

讨论中我提了两个问题：

> "Singleton 和多线程有什么关系？"
> "多个 HTTP 请求调同一个 Singleton，是不是得排队？"

**答案**：

1. Singleton 只保证"对象唯一"，**不限制线程数**。多线程可以同时进入同一个 Singleton 的方法。
2. **不会排队**。HTTP 请求本身是多线程的，三个请求同时调同一个 Singleton 的方法，三个线程并行执行。只有加了 `lock` 才会排队。

```csharp
// 没锁：多线程同时进，不排队
public int Add(int a, int b) {
    return a + b;  // 纯计算，各算各的
}

// 有锁：排队
private int _counter;
public int NextId() {
    lock (this) {          // 线程B等线程A用完才能进
        return _counter++;
    }
}
```

## 三、线程安全的关键——共享可变状态

### 3.1 四种情况

```
                     无共享状态                    有共享状态
              ┌─────────────────────┬─────────────────────────┐
  Singleton   │ ① 安全 ✅            │ ② 需要加锁 ⚠️            │
              │ 纯计算方法、无字段     │ _counter++、List.Add()  │
              │ 1000个线程同时调都行   │ 多线程同时写就崩          │
              ├─────────────────────┼─────────────────────────┤
  多个实例    │ ③ 安全 ✅            │ ④ 也可能不安全 ⚠️        │
              │ 每个线程自己的实例     │ static 字段共享、        │
              │ 天然隔离              │ 同一行数据库并发写        │
              └─────────────────────┴─────────────────────────┘
```

核心结论：**线程安全的关键是"有没有共享的可变状态"，而不是实例数量。**

### 3.2 具体例子

**① Singleton，无状态 → 安全**（最常见的工具类）：
```csharp
public class Calculator {
    public int Add(int a, int b) => a + b;
}
// 1000 个线程同时调，互不影响
```

**② Singleton，有状态 → 危险**：
```csharp
public class Counter {
    private int _count = 0;
    public int Next() => _count++;  // 读→加→写不是原子的
}
// 线程A读到0，线程B也读到0，都写成1 → 丢了一个计数
```

**④ 多实例，但共享数据库行 → 同样危险**：
```csharp
// 每个请求有自己的 UserService 实例（Scoped）
public class UserService(AppDbContext db) {
    public async Task Deduct(long id) {
        var user = await db.Users.FindAsync(id);
        user.Balance -= 100;
        await db.SaveChangesAsync();  // 请求A和B同时操作同一行！
    }
}
```
每个请求有自己的实例，但操作同一行数据库记录——还是需要乐观锁或事务保护。

## 四、实际选型：Singleton vs Scoped

### 4.1 速查表

| 服务类型 | 选什么 | 为什么 |
|---------|--------|--------|
| 数据库查询 `QueryService` | `Scoped` | 需要 DbContext，和请求同生共灭 |
| 业务逻辑 `AttendanceRecordService` | `Scoped` | 依赖 DbContext + 多个 QueryService |
| HTTP 客户端 `HttpClient` | `Singleton` | 全局复用连接池，不依赖请求 |
| 配置读取 `IConfiguration` | `Singleton` | 配置加载一次就够了 |
| 缓存 `IMemoryCache` | `Singleton` | 全局共享缓存，天然的 |
| 日志 `ILogger<T>` | `Singleton` | 无状态，纯输出 |
| 密码哈希、JWT 生成 | `Singleton` | 无状态工具类 |

### 4.2 决策口诀

> 需要 DbContext → **Scoped**
> 纯计算 / 无状态 / 全局共享 → **Singleton**

### 4.3 这次实际改了什么

只改了 Attendance 模块的 3 个文件（SiteManagement 因其被多模块调用，保守回退了）：

```diff
// 注册方式
- builder.Services.AddSingleton<IAttendanceOnlinePeriodQueryService, ...>();
+ builder.Services.AddScoped<IAttendanceOnlinePeriodQueryService, ...>();

// 构造函数：不再通过 IServiceProvider 手动借连接
- public AttendanceOnlinePeriodQueryService(IServiceProvider serviceProvider)
+ public AttendanceOnlinePeriodQueryService(AppDbContext dbContext)

// 方法内：不再 CreateAsyncScope()
- await using var scope = serviceProvider.CreateAsyncScope();
- var ctx = scope.ServiceProvider.GetRequiredService<AppDbContext>();
- return await ctx.HelmetVsPersons.Where(...).ToListAsync();
+ return await dbContext.HelmetVsPersons.Where(...).ToListAsync();
```

改之前每人 5 个 scope（chunk 外层 + 4 次方法内借），改之后每人只有 chunk 的 scope 一个，连接复用量级大幅下降。

## 六、总结

今天从一个具体性能问题出发，把 Singleton/Scoped 的本质区别、线程安全的真正含义、以及实际项目中的选型框架都理了一遍。最重要的收获是：

**"线程安全"不等于"Singleton 才需要关心"**。Singleton 只是对象唯一，真正危险的是共享可变状态——无论是全局单例里的字段、static 变量、还是多请求并发写的同一行数据库记录。

下一步应该重点学习**并发控制**（锁、线程池、连接池），才能在优化接口性能时做出有理有据的决策。
