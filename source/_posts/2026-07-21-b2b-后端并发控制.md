> 并发选型时大多数人问的是"乐观锁和悲观锁哪个好"——这是个伪问题。该问的是"你的场景长什么样"。这篇文章用一张总表覆盖所有场景，再按场景逐一拆解"为什么"，最后收敛成一个判断流程。

---

## 总表

| 场景 | 典型系统 | QPS | 冲突率 | 核心风险 | 推荐方案 | 不要用 |
|------|---------|:---:|:-----:|---------|---------|--------|
| **B2B 管理后台** | OA、ERP、运营后台、库存管理 | <100 | 极低 | 慢SQL打满CPU、长事务持锁、编辑冲突 | 原子UPDATE / 乐观锁 + 前置校验 + 分批提交 | Redis锁、消息队列 |
| **B2B 前台交易** | 企业采购平台、B2B电商 | 数千~数万 | 中 | 大促热点行冲突 | 乐观锁+退避 / Redis守门员+DB安全网 | 纯Redis锁不落DB |
| **C端高并发写** | 秒杀、抢票、闪购 | 数千~10万+ | 极高 | 超卖、热点行瓶颈 | Redis Lua预扣 → MQ异步落库 → DB CHECK | 纯DB行锁扛量 |
| **支付 / 金融** | 支付回调、转账 | 视业务 | — | **重复扣款**（外部副作用） | 幂等键 + DB唯一约束 | 任何分布式锁 |
| **状态机流转** | 订单、审批、工单 | 视业务 | 极低 | 状态跳跃 | 乐观锁 + `WHERE status=? AND version=?` | 悲观锁 |
| **任务队列消费** | 异步worker、定时Job | — | 中 | 重复消费、重试风暴 | 幂等 + SKIP LOCKED + 死信 + 退避抖动 | 无幂等重试、固定间隔退避 |
| **PG运维** | 所有用PG的系统 | — | — | 长事务阻塞autovacuum | `idle_in_transaction_session_timeout` + 监控死元组 | 事务里等人 |

## 按场景拆解

### 1. B2B 管理后台

**长什么样**：几十到几百个操作员，QPS 不到三位数，但单次操作很重——复杂报表查询、批量导入导出、编辑已消耗的流水。

**为什么是这个方案**：QPS 低意味着 DB 行锁完全够用，不需要 Redis；但操作重意味着慢 SQL 和长事务才是真正杀手。库存扣减用 `UPDATE ... WHERE stock >= @qty` 一条语句就够——不先 SELECT 再 UPDATE，判断和扣减合一，数据库行锁自然串行化并发请求。编辑冲突用前置校验而非 CHECK 约束兜底——用户应该看到"该记录已被他人修改"而不是 500。

**为什么不要用 Redis 锁**：QPS <100，DB 行锁绰绰有余。Redis 是额外的运维负担，且库存扣减必须和流水写入同一个 DB 事务——Redis 和 DB 跨不了事务边界。

---

### 2. B2B 前台交易

**长什么样**：企业采购员在大促期间集中下单，用户量比管理后台大两个数量级，有热点商品冲突。

**为什么是这个方案**：冲突率中等（不像秒杀那么极端），乐观锁+退避通常够用。如果热点集中，加一层 Redis 锁做守门员降低 DB 压力，但 **DB CHECK 约束仍然是最后安全网**——Redis 锁会过期，只有 DB 能保证不超扣。

**和 B2B 管理后台的关键区别**：用户量级差两个数量级，所以"有没有必要加 Redis"的答案不同。后台不用，前台可能需要。判断标准是 DB CPU 是否持续告警且慢查询里库存 UPDATE 占比高——信号出现之前不要加。

---

### 3. C 端高并发写

**长什么样**：秒杀、抢票、抢券。同一时刻数万人抢同一个商品，冲突率极高。

**为什么是这个方案**：冲突率高到乐观锁的重试代价不可接受，纯 DB 行锁吞吐不够。必须是三层：Redis Lua 原子预扣抗量（QPS ~10万）→ 成功的进 MQ 异步落库 → DB CHECK 兜底。每一层都是下一层的守门员。

**和 B2B 前台交易的关键区别**：冲突率从"中等"变成"极高"，方案从"可能加一层 Redis"变成"必须三层"。量的差距会导致质的方案变化。

---

### 4. 支付 / 金融

**长什么样**：支付回调、转账、扣款——涉及外部系统实际资金变动。

**为什么是这个方案**：支付有外部副作用。假设你用分布式锁保护支付接口，锁超时了，请求重试——银行已经扣了款，你不知道，再扣一次。分布式锁保护的是"并发不重复调用"，但它保护不了"超时后的重试"。

**唯一正确的做法**：给每个支付请求分配幂等键（`request_id`），DB 加唯一约束。同一个 request_id 第二次来，查 DB 直接返回已有结果，绝不再次调用外部接口。外部副作用不可回滚，这是铁律。

---

### 5. 状态机流转

**长什么样**：订单状态（待支付→已支付→已发货→已完成）、审批状态流转。

**为什么是这个方案**：冲突率极低——同一张订单同时被两个人改的概率很小。而且天然幂等：`UPDATE orders SET status='paid' WHERE id=? AND status='pending'`，如果订单已经不是 pending，SQL 自然不生效。乐观锁 version 列足矣，加悲观锁是浪费。

---

### 6. 任务队列消费

**长什么样**：多个 worker 实例抢同一个队列里的任务。

**为什么是这个方案**：
- **幂等**是第一优先级——消息队列承诺"至少投递一次"，不承诺"只投递一次"。消费者的代码必须假设同一条消息会被投递多次。
- **SKIP LOCKED** 让 worker 跳过已被同伴锁定的任务行，不阻塞等待。
- **退避加抖动**是因为固定间隔重试会让所有失败消息在同一时刻同时重试，形成重试风暴。指数退避拉大间隔，随机抖动打散时间点。
- **死信队列**收容重试 N 次仍失败的消息——不是 bug，是容错体系承认"有些错误自动化处理不了"。

---

### 7. PostgreSQL 运维

**长什么样**：所有依赖 PG 的系统都可能踩。

**为什么是这个方案**：PG 的 autovacuum 负责清理 UPDATE/DELETE 产生的死元组。但 autovacuum 只能清理"在所有活跃事务开始之前就死了"的元组。如果有一条事务跑了 5 分钟没提交，这 5 分钟内产生的所有死元组都清理不掉——堆积 → 索引退化 → CPU 打满。这不是并发问题，是一条长事务引发的雪崩。

**监控比方案更重要**：`SELECT relname, n_dead_tup, last_autovacuum FROM pg_stat_user_tables WHERE n_dead_tup > 10000`。死元组数量比 QPS 更能反映系统健康。

**最常见的触发方式**：在事务里等用户操作。查询完就关连接，保存时开新事务——这条规则救过很多系统。

## 判断流程

拿到一个新场景，按这个顺序走：

```
1. 有没有外部副作用？
   ├── 有 → 支付/金融那一行。忘掉锁，先做幂等。
   └── 没有 → 往下

2. QPS 大概多少？操作轻还是重？
   ├── QPS 低 + 操作重 → B2B管理后台那一行。原子UPDATE/乐观锁，别上中间件。
   ├── QPS 中 + 有热点 → B2B前台交易那一行。乐观锁+退避，按需加Redis守门员。
   └── QPS 极高 + 冲突极高 → C端高并发写那一行。三层：Redis → MQ → DB。

3. 冲突率极低？
   └── 状态机那一行。乐观锁，完事。

4. 不是请求/事务，是后台任务？
   └── 任务队列消费那一行。先做幂等，再做退避抖动。

5. 以上都不是，排查数据库本身？
   └── PG运维那一行。盯死长事务和死元组。
```



## 一句话

**先定位场景，再选方案，别反过来。** 大部分并发选型错误，不是因为不会用某个锁，而是把场景认错了。

下面以 B2B 管理后台里最典型的**物品进出库系统**为例，逐一拆解这个场景下会遇到的具体问题和解法。
> 需求：
> 1.保证单个仓库在并发环境下 库存一致 =>（并发数据一致性问题）
> 2.保证数据可回溯 =>（归档数据如何归档问题）

---

## 问题一：出库并发，怎么不超扣？

多个操作员可能同时从同一个仓库申领同一种物品。最简单的"先查再扣"有竞态——SELECT 和 UPDATE 之间另一个操作员也查到了同样的库存数，两人都扣减成功，加起来超了。

**正确做法：一条 SQL，判断和扣减合一。**

```sql
-- 全局库存原子扣减
UPDATE inventory_ledger
SET total_quantity = total_quantity - @qty
WHERE project_id = @p AND material_id = @m AND total_quantity >= @qty;

-- 仓库维度原子扣减（同一事务内）
UPDATE warehouse_stock
SET stock_quantity = stock_quantity - @qty
WHERE project_id = @p AND warehouse_id = @w
  AND material_id = @m AND stock_quantity >= @qty;
```

`affected rows == 0` → 回滚，返回"库存不足"。再加 `CHECK (total_quantity >= 0)` 兜底。

**为什么不用 SELECT FOR UPDATE？** 多一次 DB 往返。**为什么不用乐观锁 version 列？** 需要先 SELECT 再 UPDATE，两次往返，冲突了还要重试循环。**为什么没用 Redis？** QPS 不到 100，DB 行锁绰绰有余，而且扣减必须和流水写入同一个 DB 事务，Redis 跨不了事务边界。

**多仓库场景**同样需要注意：同一种物品分布在不同仓库，全局总量够不代表具体仓库够。两个 UPDATE 在同一事务内，全部成功才提交。

---

## 问题二：编辑已发生后续操作的流水

库存是物化存储的——出入库时增量更新汇总表，查询直接读物化值。代价是编辑和删除流水时必须回滚库存。

**但回滚前必须先做前置校验。** 比如一个入库单录入了 100 件，之后已被出库消耗了 30 件，当前库存只有 70。如果直接回滚入库（库存 -100），会变成负数：

```
事务内：
  1. 加载旧流水（入库 100 件到 A 仓库）
  2. 前置校验：当前 A 仓库 stock=70 < 100？
     → 是 → 返回友好提示："该入库已被后续出库消耗，无法编辑"
     → 不依赖 CHECK 约束抛 500 给用户看
  3. 校验通过 → 回滚旧库存 → 应用新库存 → Commit
```

**这是管理后台最重要的防御性编程习惯**：CHECK 约束是最后安全网，不是用户交互层。用户应该看到业务提示，不是数据库错误码。

---

## 问题三：软删除怎么兼顾审计和性能

审计要求删除操作可追溯，但活动表加 `is_deleted` 字段会让所有查询都带过滤条件。

**方案：归档表 + JSON 快照 + 物理删除。**

```
事务内：
  1. 前置校验（同上）
  2. 回滚库存
  3. 序列化流水为 JSON → INSERT archive 表
  4. DELETE 原表（明细级联删除）
  5. Commit
```

活动表查询干净，归档表满足审计追溯——不需要 `is_deleted`。

---

## 问题四：批次号生成——进程内的并发也别过度设计

出入库流水需要生成批次号（前缀 + 日期 + 流水号）。多个请求线程可能同时抢号：

```csharp
// 数值递增：Interlocked，CPU 硬件指令，无锁
int nextSeq = Interlocked.Increment(ref _batchSeqIn);

// 日期切换时从 DB 取当天最大值初始化，lock + double-check
if (_batchDate != datePart) {
    lock (_batchLock) {
        if (_batchDate != datePart) {
            // 从 DB 取当天最大值，初始化计数器
            _batchDate = datePart;
        }
    }
}
```

初版曾加了一个 `ConcurrentDictionary` 做 DB + 内存双保险——后来删了。**单机单进程下 Interlocked 完全够用。学了新东西就想用，是过度设计最常见的来源。**

---

## 问题五：慢查询和长事务——不并发也能搞垮系统

QPS 不到 1，但系统卡死——管理后台的经典困境。

**慢查询**：列表页多表 JOIN + 聚合 + 模糊搜索，一条 SQL 跑几秒。解法是物化库存——查询直接读物化值，不实时 SUM 明细。

**长事务**：操作员打开编辑页面 → 离开了一会儿 → 回来点保存。如果这期间持有一个数据库事务，锁就持了几分钟。而且长事务阻塞 PostgreSQL 的 autovacuum → 死元组堆积 → 索引退化 → CPU 打满。

**铁律：用户操作的间隙不持有数据库事务。** 查询完就释放连接，保存时开新事务。

## 判断流程

拿到一个新场景，按顺序问：

```
1. 有没有外部副作用？
   ├── 有 → 忘掉锁，先做幂等
   └── 没有 → 往下

2. QPS 大概多少？
   ├── <100 → 原子UPDATE / 乐观锁够用，别上中间件
   ├── 中等 + 有热点 → 乐观锁+退避，按需加Redis
   └── 极高 → Redis Lua → MQ → DB CHECK 三层

3. 操作是轻还是重？
   ├── 重的（多表聚合、批量操作）→ 物化汇总 + 分批提交
   └── 轻的 → 乐观锁 version 列

4. 有没有人在事务间隙等待？
   └── 有 → 绝不在事务里等人
```
