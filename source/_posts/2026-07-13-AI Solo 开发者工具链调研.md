## 五、踩过的坑——绝对不要再踩

### 坑 1：last30days 的 SKILL.md 极长，不要跳过 Step 0.55 和 Step 0.75
- **现象**：直接跑 engine 不加 `--plan` 和 `--subreddits` 等 flag，产出是 bland 的 4-bullet 摘要
- **正确做法**：每次都跑完整的 pre-research WebSearch → 解析 subreddits/handles → 生成 JSON query plan → 写 tmpfile → 用 `--plan "$QUERY_PLAN_FILE"` 传参
- **Python 预检是必需的**：每次 engine 运行前必须解析 `LAST30DAYS_PYTHON`，否则会报 permission denied

### 坑 2：不要用 `bash -lc '...'` 包装 engine 命令
- **现象**：`-lc` 的单引号在遇到搜索词中的 apostrophe（如 `Kanye West's album`）时会提前终止，导致 `zsh: unmatched` 错误
- **正确做法**：直接运行 heredoc block，`<<'PLAN_EOF'` 标记已经使 heredoc 内容 apostrophe-safe

### 坑 3：Plan JSON 通过 tmpfile 传递，不要内联 `--plan '$JSON'`
- **现象**：内联 JSON 中的 special chars（中文字符、apostrophe、`$`）会破坏 shell 解析
- **正确做法**：`QUERY_PLAN_FILE=$(mktemp)` → `cat >| "$QUERY_PLAN_FILE" <<'PLAN_EOF'` → `--plan "$QUERY_PLAN_FILE"`

### 坑 4：不要用 WebSearch-only 输出代替 engine 运行
- **现象**：last30days 的核心价值在 Reddit 社区评论 + HN 讨论，WebSearch 只能补充博客和新闻
- **正确做法**：先跑 engine（获取社区数据），再跑 2-3 个 WebSearch 补充博客/教程/新闻

### 坑 5：不要跳过 Step 2.5（把 WebSearch 补充结果追加到 raw file）
- **现象**：保存的 raw file 在 Source Coverage 处截断，未来会话无法追溯到 WebSearch 来源
- **正确做法**：每次 engine 运行后，定位 `[last30days] Saved output to {path}` 日志中的路径，把 `## WebSearch Supplemental Results` 追加到文件末尾

### 坑 6：setup 只跑一次，但 cookie 需要用户同意
- **现象**：如果静默运行 `setup --allow-browser-cookies` 而不先征求同意，等于未经授权读取浏览器 cookie
- **正确做法**：先检查 `BROWSER_CONSENT=true` 是否已在 `.env` 中，如果不在，弹出 AskUserQuestion 征求同意

### 坑 7：GitHub SSH 认证——如果用户没配 SSH key，用 HTTPS 代替
- **现象**：`git@github.com: Permission denied (publickey)` 导致插件市场 clone 失败
- **修复**：`git config --global url."https://github.com/".insteadOf git@github.com:`
- 这个修复已经应用，理论上后续的 git clone 都会自动走 HTTPS
